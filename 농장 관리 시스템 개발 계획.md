# 난 농장 관리 시스템 MVP 개발 계획

## 1. 프로젝트 목표

비닐하우스 난 농장을 관리하기 위한 웹 기반 농장 경영 시스템을 개발한다.

농장은 총 15개 동으로 구성되어 있고, 각 동마다 3개의 배드가 존재한다. 하나의 배드에는 여러 종류의 난 묶음이 존재할 수 있다. 사용자는 위에서 농장을 내려다보는 맵 UI를 통해 동과 배드를 선택하고, 해당 구역에 어떤 난이 있는지 확인하며, 농약·비료·분갈이 등의 작업 이력을 기록할 수 있어야 한다.

또한 판매 내역은 기존에 전표로 관리되고 있으므로, 판매 전표를 디지털로 입력하고 출력할 수 있는 기능도 포함한다.

이 프로젝트의 핵심 목표는 “완벽한 자동화”가 아니라 “현장에서 꾸준히 사용할 수 있는 쉬운 관리 시스템”이다.

---

## 2. 핵심 사용자

### 부모님 세대 작업자

* IT 기기 입력에는 익숙하지 않을 수 있음
* 종이 출력물, 전표, 장부 형태를 선호
* 복잡한 입력 화면은 사용하기 어려움

### 자식 세대 관리자

* PC/태블릿/모바일을 이용해 데이터를 입력하고 관리
* 부모님이 작성한 수기 전표나 작업 내용을 시스템에 입력
* 필요 시 출력해서 부모님께 제공

따라서 시스템은 다음 원칙을 따른다.

* 입력은 최대한 단순하게 만든다.
* 버튼과 글자는 크게 한다.
* 작업 이력 입력은 10초 안에 가능해야 한다.
* 부모님이 직접 쓰지 않더라도 출력물로 확인할 수 있어야 한다.
* 처음부터 개체 단위 완전 추적을 목표로 하지 않는다.
* 1차 버전은 동 → 배드 → 난 묶음 단위 관리에 집중한다.

## 3. 농장 구조

농장은 다음 구조를 가진다.

```text
농장
 ├─ 1동
 │   ├─ 1번 배드
 │   ├─ 2번 배드
 │   └─ 3번 배드
 ├─ 2동
 │   ├─ 1번 배드
 │   ├─ 2번 배드
 │   └─ 3번 배드
 ...
 └─ 15동
     ├─ 1번 배드
     ├─ 2번 배드
     └─ 3번 배드
```

총 15개 동, 45개 배드를 기본 데이터로 생성한다.

각 동 내부에는 세로로 긴 배드 3개가 나란히 존재한다. UI 맵에서도 각 동 안에 긴 세로형 배드 3개가 보이도록 표현한다.

---

## 4. 관리 단위

초기 MVP에서는 개별 화분 하나하나를 관리하지 않는다.

관리 단위는 “난 묶음”이다.

난 묶음은 다음 조건을 만족하는 단위다.

```text
같은 배드에 위치하고,
같은 품종 또는 같은 관리 그룹이며,
같은 화분 크기/연차/상태로 묶어서 관리할 수 있는 난들
```

예시:

```text
3동 - 2번 배드
 ├─ 카틀레야 A / 4치 / 120분 / 2년차
 ├─ 카틀레야 B / 5치 / 80분 / 개화주
 └─ 덴드로비움 C / 3.5치 / 200분 / 판매 가능
```

## 5. 주요 기능 범위

## 5.1 대시보드

대시보드는 다음 정보를 보여준다.

* 전체 동 수: 15개
* 전체 배드 수: 45개
* 분갈이 예정 개수
* 최근 작업일
* 최근 농약 살포일
* 최근 비료 작업일
* 최근 분갈이 작업일
* 상태 이상이 기록된 배드 또는 난 묶음

대시보드의 중심은 농장 맵이다.

---

## 5.2 농장 맵 UI

위성사진처럼 위에서 농장을 내려다보는 형태의 가상 맵 UI를 만든다.

단, 실제 위성사진을 그대로 쓰기보다는 사용하기 쉬운 도식화된 맵으로 구현한다.

### 맵 요구사항

* 15개 동을 화면에 배치한다.
* 각 동은 긴 직사각형 카드처럼 표현한다.
* 각 동 안에는 세로로 긴 배드 3개를 나란히 표시한다.
* 각 동에는 `1동`, `2동`, ..., `15동` 라벨을 붙인다.
* 각 배드에는 `배드1`, `배드2`, `배드3` 라벨을 붙인다.
* 사용자가 동을 클릭하면 해당 동이 선택된다.
* 사용자가 배드를 클릭하면 해당 배드가 선택된다.
* 선택된 동/배드는 색상 또는 테두리로 강조한다.
* 선택된 배드의 상세 정보는 우측 패널에 표시한다.

### 줌 개념

맵은 향후 다음 단계로 확장될 수 있어야 한다.

```text
전체 보기
→ 동 보기
→ 배드 보기
→ 세부 구역 보기
```

MVP에서는 실제 복잡한 GIS 기능까지 구현하지 않아도 된다.
하지만 UI 구조상 줌 단계 개념을 고려한다.

예시:

* 기본 화면: 15개 동과 각 동의 3개 배드 표시
* 특정 배드 선택: 우측 패널에 상세 정보 표시
* 확대 상태 예시: 선택된 배드 안에 `구역1`, `구역2`, `구역3`, `구역4` 같은 세부 구역을 표시할 수 있도록 구조 설계

## 5.5 작업 이력 관리

농약, 비료, 분갈이 등의 작업 이력을 기록할 수 있어야 한다.

중요한 점은 작업 대상이 여러 수준일 수 있다는 것이다.

### 작업 대상

```text
전체 농장
특정 동
특정 배드
특정 난 묶음
```

예를 들어 농약을 1동 전체에 살포했다면, 사용자가 1동만 선택해서 작업 이력을 등록할 수 있어야 한다.

### 작업 유형

```text
농약
비료
분갈이
관수
위치 이동
병해충 발견
생육 이상
개화 확인
입고
출고
폐기
일반 메모
```

MVP에서는 우선 다음 작업 유형만 구현해도 된다.

```text
농약
비료
분갈이
상태 기록
일반 메모
```

### 작업 이력 필드

```text
id
작업 유형
작업일
대상 유형: FARM / HOUSE / BED / ORCHID_GROUP
대상 id
약제명 또는 비료명
희석 배수 또는 농도
수량
작업자
메모
등록일
수정일
```

작업 이력이 저장되면 화면에서는 난 묶음 또는 배드별로 최근 작업일을 자동 계산해서 보여준다.

사용자가 `최근 농약일`, `최근 비료일`, `최근 분갈이일`을 직접 수정하지 않도록 한다.
항상 작업 이력에서 계산한다.

## 5.7 출력 기능

부모님 세대 사용자를 위해 출력 기능이 중요하다.

```text
판매 전표 출력
```

출력물은 A4 기준으로 읽기 쉽게 만든다.
복잡한 디자인보다 글자가 크고 정보가 명확한 것이 중요하다.

7. 기술 스택 제안

이 프로젝트는 단순한 화면 MVP가 아니라 실제 농장에서 장기간 사용할 수 있는 운영 시스템을 목표로 한다.
따라서 빠른 개발도 중요하지만, 판매 전표·작업 이력·난 묶음 관리 데이터가 장기간 안정적으로 쌓일 수 있도록 프론트엔드와 백엔드를 분리한 구조로 시작한다.

권장 스택

Frontend
- Next.js
- TypeScript
- Tailwind CSS
- React Hook Form
- Zod

Backend
- Spring Boot
- Java 21
- Spring Data JPA
- Bean Validation
- Spring Security, 초기에는 간단한 로그인 수준
- PostgreSQL

Infra
- Docker Compose
- PostgreSQL
- 로컬 파일 저장소, 추후 MinIO 또는 S3 호환 스토리지 확장

이유

- Next.js는 농장 맵 UI, 전표 입력 화면, 출력 화면을 빠르게 만들기에 적합하다.
- Spring Boot는 판매 전표, 작업 이력, 난 묶음 관리처럼 도메인 로직이 쌓이는 백엔드 시스템에 적합하다.
- PostgreSQL은 실제 운영 데이터를 장기간 저장하기에 적합하며, 추후 서버 이전과 백업이 쉽다.
- Docker Compose를 사용하면 개발 환경과 농장 내부 서버 배포 환경을 비슷하게 유지할 수 있다.
- 전체 구조는 MSA가 아니라 모듈러 모놀리스로 시작한다.

## 8. 데이터 모델 초안

다음 모델을 설계한다.

```
model House {
  id        Int      @id @default(autoincrement())
  number    Int      @unique
  name      String?
  memo      String?
  beds      Bed[]
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}

model Bed {
  id           Int           @id @default(autoincrement())
  houseId      Int
  number       Int
  memo         String?
  house        House         @relation(fields: [houseId], references: [id])
  orchidGroups OrchidGroup[]
  createdAt    DateTime      @default(now())
  updatedAt    DateTime      @updatedAt

  @@unique([houseId, number])
}

model OrchidGroup {
  id          Int      @id @default(autoincrement())
  bedId       Int
  genus       String
  varietyName String
  quantity    Int
  potSize     String?
  ageYear     Int?
  status      String
  memo        String?
  bed         Bed      @relation(fields: [bedId], references: [id])
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
}

model WorkRecord {
  id             Int      @id @default(autoincrement())
  workType       String
  workDate       DateTime
  targetType     String
  targetId       Int?
  materialName   String?
  dilutionRatio  String?
  quantity       String?
  worker         String?
  memo           String?
  createdAt      DateTime @default(now())
  updatedAt      DateTime @updatedAt
}

model Customer {
  id        Int      @id @default(autoincrement())
  name      String
  ownerName String?
  phone     String?
  address   String?
  memo      String?
  slips     SalesSlip[]
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}

model SalesSlip {
  id            Int             @id @default(autoincrement())
  slipNumber    String          @unique
  saleDate      DateTime
  customerId    Int
  totalAmount   Int
  paymentStatus String
  paymentMethod String?
  memo          String?
  customer      Customer        @relation(fields: [customerId], references: [id])
  items         SalesSlipItem[]
  createdAt     DateTime        @default(now())
  updatedAt     DateTime        @updatedAt
}

model SalesSlipItem {
  id          Int       @id @default(autoincrement())
  salesSlipId Int
  itemName    String
  genus       String?
  spec        String?
  quantity    Int
  unitPrice   Int
  amount      Int
  memo        String?
  salesSlip   SalesSlip @relation(fields: [salesSlipId], references: [id])
}
```

추후 확장을 위해 `workType`, `status`, `genus`, `paymentStatus`는 enum으로 바꿀 수 있다.
MVP에서는 문자열로 시작해도 된다.

* 우측 상세 패널 표시

### 3단계: 난 묶음 관리

* 배드별 난 묶음 조회
* 난 묶음 등록
* 난 묶음 수정
* 난 묶음 삭제

### 4단계: 작업 이력 관리

* 작업 이력 등록
* 대상 유형 선택: 농장/동/배드/난 묶음
* 작업 이력 목록 조회
* 배드별 최근 농약/비료/분갈이 날짜 계산

### 5단계: 판매 전표 관리

* 거래처 관리
* 판매 전표 등록
* 판매 품목 등록
* 합계 자동 계산
* 전표 목록 조회

### 6단계: 출력

* 판매 전표 출력
* 배드별 관리표 출력
* 작업 이력 출력
* 브라우저 print CSS 적용

## 11. UI 구현 규칙

* 모바일보다는 PC/태블릿 우선으로 만든다.
* 글자 크기는 작게 하지 않는다.
* 현장 사용을 고려해 버튼을 크게 만든다.
* 색상은 흰색, 연한 회색, 초록색 중심으로 한다.
* 선택된 동은 초록색 테두리로 강조한다.
* 선택된 배드는 파란색 또는 진한 초록색으로 강조한다.
* 위험/이상 상태는 주황색 또는 빨간색으로 표시한다.
* 화면이 복잡해지면 안 된다.
* 입력 폼은 한 화면에 너무 많은 필드를 넣지 않는다.
* 자주 쓰는 작업은 버튼 클릭 수를 최소화한다.

## 12. 농장 맵 컴포넌트 요구사항

`FarmMap` 컴포넌트를 만든다.

### Props 예시

```ts
type FarmMapProps = {
  houses: HouseWithBeds[];
  selectedHouseId?: number;
  selectedBedId?: number;
  onSelectHouse: (houseId: number) => void;
  onSelectBed: (bedId: number) => void;
};
```

### 구조

```text
FarmMap
 ├─ HouseBlock x 15
 │   ├─ BedLane 1
 │   ├─ BedLane 2
 │   └─ BedLane 3
 └─ ZoomControl
```

### 시각 표현

각 동은 긴 사각형으로 표시한다.
각 동 내부에는 세로로 긴 배드 3개가 나란히 배치된다.

```text
┌─────────────┐
│ 1동         │
│ ┃ ┃ ┃       │
│ ┃ ┃ ┃       │
│ ┃ ┃ ┃       │
│ B1 B2 B3    │
└─────────────┘
```

실제 UI에서는 15개 동이 가로 방향으로 나열되고, 각 동 내부의 3개 배드가 세로로 길게 표시된다.

선택된 배드는 색으로 강조한다.

## 13. 작업 이력 계산 규칙

특정 배드의 최근 농약일을 보여줄 때는 다음 이력을 고려한다.

* 해당 배드에 직접 등록된 농약 이력
* 해당 배드가 속한 동에 등록된 농약 이력
* 전체 농장에 등록된 농약 이력

특정 난 묶음의 최근 농약일을 보여줄 때는 다음 이력을 고려한다.

* 해당 난 묶음에 직접 등록된 농약 이력
* 해당 난 묶음이 속한 배드에 등록된 농약 이력
* 해당 배드가 속한 동에 등록된 농약 이력
* 전체 농장에 등록된 농약 이력

비료와 분갈이도 같은 방식으로 계산한다.
단, 분갈이는 보통 난 묶음 또는 배드 단위가 더 자연스럽다.

## 14. MVP 완료 기준

다음이 가능하면 MVP 완료로 본다.

* 15개 동과 45개 배드가 맵에 표시된다.
* 사용자가 동과 배드를 클릭할 수 있다.
* 배드를 클릭하면 해당 배드의 난 목록이 우측에 표시된다.
* 난 묶음을 등록/수정/삭제할 수 있다.
* 농약/비료/분갈이 작업 이력을 등록할 수 있다.
* 배드별 최근 작업일이 표시된다.
* 판매 전표를 등록할 수 있다.
* 판매 품목의 금액과 총액이 자동 계산된다.
* 판매 전표를 브라우저에서 출력할 수 있다.
* 배드별 관리표를 출력할 수 있다.

## 16. Codex 작업 지시

위 내용을 기준으로 Next.js + TypeScript + Tailwind CSS + Prisma 기반의 MVP를 구현한다.

우선순위는 다음과 같다.

1. 데이터 모델과 시드 데이터
2. 농장 맵 대시보드
3. 배드 상세 패널
4. 난 묶음 CRUD
5. 작업 이력 CRUD
6. 판매 전표 CRUD
7. 출력 화면

완성도 높은 디자인보다 실제 사용 가능한 동작을 우선한다.
다만 농장 맵 UI는 이 프로젝트의 핵심이므로, 15개 동과 각 동의 3개 세로형 배드가 직관적으로 보이도록 구현한다.
