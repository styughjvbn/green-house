# Domain Rules for Codex

이 문서는 코드와 OpenAPI만 보고는 놓치기 쉬운 도메인 의도와 운영 규칙만 기록한다.  
필드 타입, request/response schema, enum 전체 목록은 `openapi.yaml` 또는 실제 코드를 확인한다.

## 1. 문서 작성 원칙

- 코드에서 확인 가능한 Controller, DTO 필드, required 여부는 이 문서에 중복하지 않는다.
- API가 실제 구현되어 있는지는 `openapi.yaml` 또는 Controller 코드를 기준으로 판단한다.
- 이 문서는 구현 의도, 예외 정책, 운영상 금지 사항을 남기는 용도다.
- 과거 API 초안에 있으나 OpenAPI에 없는 endpoint는 제안/미구현으로 취급한다.

## 2. 공통 API 규칙

- Base URL은 `/api`다.
- 성공 응답은 `data`와 `message`를 갖는 공통 래퍼 형태를 사용한다.
- 에러 응답은 `error.code`, `error.message`, `error.details` 구조를 기준으로 한다.
- 요청/응답 날짜는 ISO-8601 날짜 문자열을 사용한다.
- 실제 응답 schema 이름은 OpenAPI의 `ApiResponse*` schema를 기준으로 확인한다.

## 3. 농장 구조/현황

- 농장 구조는 `house -> physical bed -> bed zone -> orchid group` 흐름으로 해석한다.
- 농장 현황 화면은 전체 조회 성격이다. 선택 대상이 동, 물리 배드, 논리 구역 중 무엇인지에 따라 포함되는 난 묶음 범위가 달라진다.
- 최근 작업 요약은 구조 API 자체의 책임으로 넣지 않고 작업 이력 API와 조합해서 표시한다.
- 줌 단계 API는 전체 농장 보기, 물리 배드 보기, 논리 구역 보기의 단계적 렌더링을 위한 데이터다.

## 4. 난 묶음

- 난 묶음 생성 위치는 논리 구역 기준이다.
- 난 묶음 생성·수정 시 품종은 자유 입력하지 않고 `varietyId`로 선택한다.
- 화분 크기는 서버가 표준 코드로 정규화하며 임의 문자열 입력은 거부한다.
- 안전하게 해석할 수 없는 기존 화분 값은 `UNMAPPED`로 보존하고 자동 그룹에서 제외한다.
- 난 묶음 수정 API는 위치 이동을 담당하지 않는다. 위치 변경은 이동 API로 분리한다.
- 난 묶음 이동 단위는 기본적으로 묶음 전체다.
- 이동 성공 시 위치 이동 작업 이력이 자동 생성된다.
- 같은 구역으로 이동 요청하면 위치와 작업 이력을 변경하지 않고 현재 난 묶음 응답을 반환하는 정책이다.
- 구역 내 정렬은 생성/이동 시 `sortOrder` 기준으로 관리한다. 수동 reorder API는 현재 OpenAPI에 없다.
- 구조 변경은 원본 난 묶음을 결과로 덮어쓰지 않고 별도 난 묶음과 계보 관계를 남긴다.
- 계보는 원본·결과 난 묶음, 관계 유형, 원인 작업, 작업 당시 원본·결과 수량을 보존한다.
- 난 묶음 계보 조회는 해당 난 묶음의 직접 원본과 직접 결과를 각각 반환한다.

## 4-1. 난 묶음 사용자 그룹

- 사용자 그룹 소속은 난 묶음의 위치·속성·상태 변경과 독립적으로 유지한다.
- 하나의 난 묶음은 여러 사용자 그룹에 동시에 속할 수 있다.
- 구성원 해제는 삭제 대신 해제 시각을 남기고, 같은 활성 소속의 중복 추가는 한 건으로 유지한다.
- 사용자 그룹은 삭제 대신 `ARCHIVED`로 보관하며 보관 후 구성원을 추가하거나 해제할 수 없다.
- 기본 목록은 활성 그룹만 반환하고 `includeArchived=true`일 때 보관 그룹을 포함한다.

## 4-2. 난 묶음 자동 그룹

- 자동 그룹은 `varietyId + 현재 ageYear + potSizeCode`로 조회 시 계산하며 별도 저장하지 않는다.
- 수량 0, 종료·폐기·판매 완료 상태, `UNMAPPED` 화분 값은 자동 그룹에서 제외한다.
- 년생 미지정 그룹 키는 `UNSPECIFIED` 토큰을 사용한다.
- 그룹 키는 영구 식별자가 아니므로 구성원 조회 시 현재 속성에 맞는 난 묶음만 반환한다.
- 난 묶음 속성 변경 직후 다음 조회 결과와 그룹별 총수량이 달라질 수 있다.

## 5. 배드 정밀 배치

- 배치 프로필은 논리 구역의 구간과 규격, 화분 크기, 배치 모드별 수용량을 표현한다.
- 실제 배치가 존재하는 구간은 삭제하면 안 된다.
- 정밀 이동은 기존 난 묶음 이동 API에 `placementMode`, `placements`, `reorganizeDueDate` 같은 추가 정보를 포함하는 확장 형태다.
- 수용량이 변경되어 추천 당시 조건과 달라진 경우 `CAPACITY_CONFLICT` 계열 충돌로 처리한다.
- 기존 단순 이동 요청은 구간 미지정 이동으로 계속 호환되어야 한다.

## 6. 작업 이력/작업 유형

- 작업 이력은 삭제보다 보존을 우선한다.
- 실수로 생성한 작업 이력은 삭제하지 않고 취소 상태로 전환한다.
- 취소된 작업 이력은 기본 조회, 최근 작업, 통계 집계에서 제외한다.
- 작업 이력 수정/삭제 API는 현재 OpenAPI에 없다. 구현 전에는 존재한다고 가정하지 않는다.
- 작업 유형은 seed 기본 유형과 사용자 커스텀 유형을 함께 사용한다.
- 비활성 작업 유형은 기존 이력 조회에는 남아야 하지만 일반 작업 기록 생성에는 사용할 수 없다.
- 시스템 작업 유형, 특히 위치 이동 계열은 일반 생성 폼에서 선택하지 않는다.
- 입고 유형은 입고 관리 흐름에서 자동 생성한다. 포트 작업은 일반 즉시 작업 기록 폼에서는 제외하고 포트 대기 입고 기록을 대상으로 실행한다. 입고 관리의 즉시 포트 작업도 완료된 `WorkOperation`으로 기록하며 별도 포트 작업 `WorkRecord`를 중복 생성하지 않는다.
- 같은 입고 기록은 활성 포트 작업 계획에 중복 포함할 수 없다. 입고 관리에서 즉시 실행할 때 활성 계획이 있으면 새 작업을 만들지 않고 기존 계획의 해당 대상을 실행한다.
- 포트 작업 결과는 하나 이상의 난 묶음이며 각 결과는 수량과 논리 구역 배치를 독립적으로 가진다. 입고 기록의 `createdOrchidGroupId`는 호환용 대표 결과이고, 전체 결과는 작업 효과의 `createdOrchidGroupIds`와 각 난 묶음의 입고 출처 연결로 추적한다.
- 작업 유형 템플릿은 입력 필드와 목록/상세 표시값을 결정한다.
- 작업 이력의 공통 검색 필드는 정형 컬럼으로 유지하고, 작업별 가변 정보는 `details`에 보관한다.
- 신규 `WorkOperation`은 상위 범위를 저장할 때 실제 난 묶음 대상과 당시 위치·속성·수량을 확정한다.
- 상위 작업은 난 묶음별 이력 행으로 복제하지 않고 대상 연결을 통해 하위 이력에 표시한다.
- 기간 작업 대상 선택은 동 전체, 자동 그룹 키, 활성 사용자 그룹, 직접 난 묶음 ID 목록을 지원한다.
- 즉시 완료 기록형 작업은 농장 전체, 동, 물리 배드, 논리 구역, 난 묶음 범위를 지원한다.
- 사용자 그룹에서는 현재 작업 가능한 구성원만 대상이 되고, 직접 선택에 작업 불가능한 난 묶음이 섞이면 요청을 거부한다.
- 자동 그룹과 사용자 그룹 구성이 저장 후 바뀌어도 확정된 작업 대상 스냅샷은 변경하지 않는다.
- 일반 기록형 작업과 자리 이동·분갈이·분주·합식·폐기는 난 묶음 대상 기간 작업으로 계획할 수 있다. 입고 포트 작업은 입고 기록 대상을 별도로 확정한다.
- 일부 대상 또는 일부 수량이 이미 실행된 작업도 취소할 수 있다. 이미 완료된 대상과 적용된 작업 효과·이력은 보존하고, 미완료 대상과 부분 완료 대상의 잔여 수량만 취소한다.
- 자리 이동 계획은 생성 시 위치를 바꾸지 않고 대상별 완료 시 목적 논리 구역과 구간을 검증한 뒤 이동 효과를 적용한다.
- 분갈이·입고 포트 계획은 생성 시 구조를 바꾸지 않고 대상별 완료 시 결과 입력을 검증한 뒤 구조 변경 효과를 적용한다.
- 분갈이·분주·합식은 실제 원본 N개와 결과 M개를 허용하고 작업 유형에 따라 `REPOTTED_TO`, `SPLIT_TO`, `MERGED_TO` 계보를 기록한다.
- 구조 변경은 여러 실행 회차에서 계획 수량 일부를 처리할 수 있으며 누적 작업 수량과 잔여 수량을 보존한다.
- 폐기 수량은 현재 가용 수량을 초과할 수 없다. 일부 폐기는 난 묶음의 기존 상태를 유지하고, 전량 폐기는 수량 0·`폐기` 상태로 전환한다.
- 결과의 `DIVIDE_CANDIDATE`와 `HELD` 구분은 각각 분주 예정과 별도 보관 상태로 생성하되 자동 그룹 계산과 계보 조회 대상에서 제외하지 않는다.
- 기능 플래그 활성화 시 사용자 수동 `WorkRecord` 생성은 거부하며, 기존 이력 조회·취소와 시스템 감사 기록은 유지한다.
- 대상별 처리는 전체 작업이 `IN_PROGRESS`일 때만 가능하고 `PAUSED`에서는 금지한다.
- 대상 완료는 작업 효과 handler 실행, 효과 감사 기록, 대상 완료 상태 변경을 같은 트랜잭션에서 처리한다.
- 완료 요청을 재시도해도 이미 적용된 효과와 최초 작업자·결과 상세를 변경하지 않는다.
- 다중 난 묶음 생성은 원본 대상 없이 완료된 작업으로 기록하며 생성 결과를 작업 효과에 연결한다.
- 같은 다중 생성 `idempotencyKey`를 재요청하면 기존 작업과 생성 결과를 반환한다.
- 다중 생성 행은 활성 품종, 표준 화분 크기, 양수 수량, 유효하고 겹치지 않는 배치 범위를 만족해야 한다.
- 다중 생성 취소 가능 여부는 생성 결과의 다른 작업, 이동 이력, 판매·재고 이동, 입고 연결을 모두 검사한다.
- 연결된 운영 데이터가 하나라도 있으면 생성 취소를 허용하지 않고 모듈별 차단 사유를 반환한다.
- 취소 가능한 다중 생성 결과는 물리 삭제하지 않고 수량 0과 `생성 취소` 상태로 전환한다.
- 다중 생성 취소는 사용자 그룹 소속 해제, 난 묶음 상태 변경, 작업·효과 취소 기록을 한 트랜잭션에서 처리한다.
- 분갈이 투입 수량은 결과 수량 합계와 손실 수량의 합과 같아야 하며, 투입 수량만큼 원본에서 차감한다.
- 원본 수량 전체를 투입하면 원본은 수량 0·`종료` 상태가 되고, 일부만 투입하면 잔여 수량으로 활성 상태를 유지한다.
- 분갈이 결과는 원본 품종과 상태를 이어받고 새 위치·년생·화분 크기를 가지며 `REPOTTED_TO` 계보로 연결한다.
- 손실 수량이 있으면 손실 사유가 필수다. 결과에 상속할 사용자 그룹은 원본의 활성 소속 중에서 명시적으로 선택한다.
- 완료된 구조 변경 작업은 직접 수정·삭제하지 않고 별도의 `CORRECTION` 작업과 보정 사유를 연결한다.
- 보정 관계가 생성되면 원본 작업 상태는 `CORRECTED`가 되며 원본 결과 난 묶음과 감사 기록은 그대로 보존한다.
- 보정 효과는 원본 구조 변경 작업이 만든 결과 난 묶음의 수량·상태만 변경할 수 있다. 다른 난 묶음은 보정 대상으로 지정할 수 없다.
- 후속 작업·이동 이력·판매·재고 이동·입고가 연결된 결과는 보정을 차단하고, 보정 수량은 현재 예약 수량보다 작을 수 없다.
- 보정 전후 수량·상태는 작업 효과 감사 상세에 보존한다. 새 난 묶음 생성이나 계보 교체는 아직 보정 효과 범위가 아니다.
- 전체 완료 전 모든 대상은 `COMPLETED` 또는 `SKIPPED`여야 한다.
- 완료된 대상이 있는 전체 작업은 일반 취소할 수 없다.
- 난 묶음 통합 이력은 직접 대상 기존 `WorkRecord`와 신규 대상 연결을 합친다. 과거 상위 범위 기존 이력은 현재 위치로 역산하지 않는다.

## 6-1. 품종/입고/자재

- 품종과 자재는 운영 데이터 보존을 위해 삭제보다 비활성화를 우선한다.
- 품종은 `속 + 품종명` 조합을 기준으로 식별하며, 같은 조합은 중복 등록할 수 없다.
- 품종 삭제는 연결된 난 묶음이나 입고 기록이 전혀 없을 때만 허용한다.
- 자재 삭제는 현재 연결 데이터가 없을 때만 허용한다.
- 품종, 자재, 입고 목록 API는 서버 페이지네이션을 사용한다. `page`는 0부터 시작하고 `size`는 기본 10, 최대 100이다.
- 입고는 `FLASK_SEEDLING`과 즉시 배치 계열이 입력 규칙과 후속 흐름이 다르다.
- `FLASK_SEEDLING`은 포트 작업 전까지 임시보관 흐름을 가진다.
- 난 묶음이 이미 생성된 입고 기록은 취소할 수 없다.
- 취소된 입고 기록만 삭제할 수 있다.
- 취소된 입고 삭제 시 자동 생성된 입고 작업 이력은 함께 정리한다.
- 일반 입고 삭제는 허용하지 않고, 취소된 입고에 한해서만 삭제를 허용한다.
- 자재 코드는 서버가 순번 기반으로 자동 발급한다.

## 7. 판매 전표

- 판매 전표 품목 금액은 `quantity * unitPrice`로 계산한다.
- 전표 `totalAmount`는 품목 금액 합계로 계산한다.
- 전표 번호는 판매일 기준 순번으로 자동 생성하는 정책이다.
- 판매 전표 품목은 하나 이상의 난 묶음 allocation을 가져야 한다.
- allocation 합계는 판매 품목 수량과 같아야 한다.
- allocation 대상 난 묶음의 품종은 판매 품목명과 같아야 한다.
- 전표 저장 시 난 묶음 `reservedQuantity`를 증가시킨다.
- `PUT /api/sales-slips/{salesSlipId}`는 `작성중` 일반 판매 전표만 수정할 수 있다.
- 현재 수정은 기존 품목 개수를 유지하는 경우만 지원한다.
- `출고 완료`, `출하 완료`로 상태 변경될 때만 실제 난 묶음 `quantity`를 차감한다.
- 판매 전표 취소는 삭제가 아니라 `salesStatus=취소` 상태 변경으로 처리한다.
- `작성중` 취소는 예약만 해제하고, 완료 전표 취소는 실제 차감 수량을 복구한다.
- 입금 이력이 있는 일반 판매 전표는 취소할 수 없다.
- 판매 전표 출력 API는 데이터를 변경하지 않고 A5 출력 화면에 필요한 조회 데이터만 제공한다.
- 브라우저 인쇄 CSS와 A5 레이아웃 렌더링은 프론트엔드 책임이다.
- 판매 상태 변경 API는 `PATCH /api/sales-slips/{salesSlipId}/sales-status`다.

## 8. 경매 출하 전표

- `salesType=AUCTION`인 판매 전표는 경매장 거래처를 기준으로 생성한다.
- 경매 판매는 `작성중`으로 저장 가능하고, `출하 완료` 시 shipment, lot가 생성된다.
- lot별 품목은 출하 수량과 0원 단가로 구성한다.
- 경매 판매 품목도 일반 판매와 같은 allocation 검증과 예약 규칙을 따른다.
- 일반 판매는 `salesType=DIRECT`이며 기존 요청처럼 유형을 생략해도 된다.
- 경매 판매전표 입금은 전표가 아니라 경매 정산에서 처리한다.
- 경매 결과나 정산에 연결된 경매 전표는 취소할 수 없다.

## 9. 출하·경매 추적

- lot 목록은 서버 페이지네이션을 사용한다. `page`는 0부터 시작하고 `size`는 기본 20, 최대 100으로 본다.
- `reviewOnly`는 매칭 실패, 수량 불일치, 반환 추정 등 운영자 확인 대상 조회다.
- `returnOnly`는 반환 추정 대상, `waitingOnly`는 재경매 대기 lot 조회다.
- 요약 API는 페이지와 무관한 전체 데이터 집계다. 전체 lot 엔티티를 모두 적재하지 말고 DB 집계 쿼리를 우선한다.
- 반환 확인은 재경매 대기, 반환 추정, 부분 반환 상태에서만 가능하다는 정책이다.
- 반환 확인 수량이 현재 확인 가능 수량보다 적으면 부분 반환, 같으면 반환 완료로 변경하고 상태 이력을 생성한다.
- 반환 수량을 생략하면 현재 확인 가능한 수량 전체를 처리한다.
- 수량 보정 시 판매 + 대기 + 반환 수량은 출하 수량과 같아야 한다.
- 상태 변경은 상태, 변경 사유, 작업자, 메모를 받고 상태 이력을 생성하는 성격이다.

## 10. 경매 정산

- 경매 정산은 경매장과 경매일 조합으로 한 건만 생성한다.
- `amount > 0`인 경매 결과만 정산 행으로 연결한다.
- 정산 재구성 요청은 기존 행을 중복 생성하지 않아야 한다.
- 현재 수수료와 공제액은 0원으로 본다.
- 예상 입금일은 거래처 정산 설정의 지연일과 달력일/영업일 계산 방식으로 정한다.

## 11. 거래처 정산 설정

- 설정이 없으면 최초 조회 시 기본 설정을 생성한다.
- 경매장은 `AUCTION_DATE`, 다른 거래처는 `SALES_SLIP`이 기본 정산 단위다.
- 입금자명 후보는 공백 제거 후 중복을 제거한다.
- `ruleJson`은 경매장별 결과 수신/파싱 규칙 확장용이며 현재 자동 수신 기능에서는 사용하지 않는다.

## 12. 수동 입금/입금 이벤트/잔액

- 입금 확인은 `PAYMENT_RECEIVED`와 `MANUAL_MATCH_CONFIRMED` 이벤트를 함께 생성하는 정책이다.
- 잔액보다 작은 금액은 부분입금, 잔액과 같은 금액은 완납으로 처리한다.
- 잔액 초과 입금은 예치금 기능이 완전히 도입되기 전까지 거절한다.
- 예치금, 계좌 자동 매칭, 자동 정산, 입금 배분은 설계상 예정되어 있지만 OpenAPI에 없는 endpoint는 아직 구현으로 보지 않는다.
- 정산/입금 이력은 나중에 추적 가능해야 하므로 직접 삭제/수정보다 보정 이벤트를 우선한다.
