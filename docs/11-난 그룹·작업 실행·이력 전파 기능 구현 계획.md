# 난 그룹·작업 실행·이력 전파 기능 구현 계획

## 1. 목표

다음 요구사항을 하나의 일관된 작업 모델로 구현한다.

```text
1. 자동 그룹
   품종 + 년생 + 화분 크기가 같은 난 묶음을 실시간으로 그룹화

2. 사용자 그룹
   사용자가 여러 난 묶음을 임의로 묶고 지속적으로 관리

3. 상위 작업의 하위 전파
   농장·동·배드·구역·그룹에 등록한 작업이
   실제 대상이 된 하위 난 묶음의 이력에도 표시

4. 기간 작업
   하루 이상 이어지는 작업을 하나의 작업 단위로 관리
   대상 난 묶음별 진행 상태 관리

5. 작업 결과 자동 반영
   분갈이·포트 작업·다중 등록 등 작업 종류에 따라
   난 묶음 생성·종료·수량 변경·위치 변경을 자동 처리

6. 이력 보존
   작업 당시 대상과 변경 전후 상태를 보존
   이후 위치나 속성이 바뀌어도 과거 이력이 달라지지 않음
```

---

# 2. 구현 원칙

## 2.1 그룹은 작업 대상을 선택하는 수단이다

자동 그룹이나 사용자 그룹 자체를 작업 이력의 최종 대상으로 사용하지 않는다.

```text
그룹 선택
→ 현재 포함된 난 묶음 조회
→ 실제 대상 난 묶음 확정
→ 대상 스냅샷 저장
→ 작업 실행
```

그룹 구성이 나중에 변경되어도 과거 작업 대상은 바뀌지 않아야 한다.

## 2.2 상위 작업 이력을 하위에 복제하지 않는다

3동 전체에 농약 작업을 등록했다고 해서 난 묶음별로 동일한 작업 이력 행을 생성하지 않는다.

```text
작업 1건
+
실제 대상 난 묶음 연결 N건
```

난 묶음 이력 조회 시 연결된 작업을 함께 조회하여 하위로 전파된 것처럼 표시한다.

## 2.3 작업 당시 대상을 저장한다

현재 위치를 기준으로 과거 작업 대상을 다시 계산하지 않는다.

```text
작업 당시 3동에 있던 난 묶음 A
→ 이후 5동으로 이동
→ A의 이력에는 과거 3동 작업이 계속 표시
```

## 2.4 구조 변경 작업은 기존 난 묶음을 직접 덮어쓰지 않는다

분갈이, 분주, 병합처럼 생육 단위가 바뀌는 작업은 원본과 결과 난 묶음을 별도로 보존한다.

```text
원본 난 묶음
→ 작업
→ 결과 난 묶음
```

원본과 결과 사이에는 계보 관계를 기록한다.

## 2.5 완료된 구조 변경 작업은 삭제하지 않는다

분갈이 완료 후 새 난 묶음이 만들어졌다면 작업 이력을 삭제하거나 기존 행을 직접 되돌리지 않는다.

잘못된 결과는 보정 작업으로 처리한다.

# 3. 먼저 확정할 정책

개발 전에 다음 정책을 문서로 확정한다.

## 3.1 자동 그룹 기준

초기 기준:

```text
variety_id
age_year
pot_size_code
```

포함 조건:

```text
활성 난 묶음
quantity > 0
```

제외 조건:

```text
종료
폐기
판매 완료
수량 0
```

화분 크기는 자유 문자열을 그대로 사용하지 않는다.

```text
3.5치
3.5"
3.5인치
105mm
```

같은 규격이 여러 문자열로 저장되어 있다면 자동 그룹이 분리되므로, `pot_size_code` 또는 화분 규격 마스터를 먼저 마련한다.

## 3.2 작업 대상 확정 시점

권장 정책:

```text
작업 등록
→ 대상 미리보기
→ 저장 시 대상 스냅샷 생성
→ 작업 시작 전 대상 새로고침 가능
→ 작업 시작 후 자동 변경 금지
```

작업 중 새로 자동 그룹에 들어온 난 묶음은 자동으로 추가하지 않는다.

필요한 경우 사용자가 직접 `대상 추가`를 실행한다.

## 3.3 사용자 그룹 소속 유지

난 묶음의 년생, 화분 크기, 위치가 변경되어도 사용자 그룹에서는 자동으로 제외하지 않는다.

```text
자동 그룹:
속성에 따라 자동 재분류

사용자 그룹:
사용자가 제거하기 전까지 유지
```

## 3.4 새 난 묶음의 사용자 그룹 상속

분갈이 결과로 새 난 묶음이 만들어졌을 때 사용자 그룹 소속은 자동 상속하지 않는 것을 기본값으로 한다.

완료 화면에서 상속할 그룹을 선택한다.

## 3.5 작업 유형별 전파 정책

작업 유형에는 다음 정책을 둔다.

```text
NONE
하위로 전파하지 않음

TARGET_SNAPSHOT
작업 당시 하위 난 묶음에 적용

LINEAGE_DESCENDANTS
결과 난 묶음의 생육 계보에서도 조회 가능
```

예시:

| 작업 유형 |     하위 대상 전파 |      결과 계보 전파 |
| ----- | -----------: | ------------: |
| 농약    |           적용 |         적용 가능 |
| 비료    |           적용 |         적용 가능 |
| 상태 기록 |           적용 |         적용 가능 |
| 동 청소  |     적용 여부 선택 |           미적용 |
| 위치 이동 |       직접 대상만 |           미적용 |
| 분갈이   |           적용 | 결과 생성 작업으로 표시 |
| 일반 메모 | 작업 유형 설정에 따름 |        기본 미적용 |

사용자 정의 작업 유형은 초기에는 기록형만 허용한다. 난 묶음 생성이나 수량 변경 같은 구조 변경 기능은 시스템 작업 유형에만 허용한다.


이 식별 키는 현재 조회 조건을 표현할 뿐, 영구적인 데이터 ID로 취급하지 않는다.

# 5. 기존 작업 이력 마이그레이션 전략

기존 `work_records`를 바로 삭제하거나 모두 새 구조로 변환하지 않는다.

## 5.1 1차 운영 방식

```text
기존 work_records
- 읽기 유지
- 신규 작성 중단 예정
- 과거 기록 보존

신규 work_operations
- 새 작업부터 사용
- 대상 스냅샷과 실행 상태 관리
```

작업 이력 조회 API는 두 데이터를 합쳐 반환한다.

```text
기존 작업 이력
+
신규 작업 실행 이력
```

## 5.2 기존 데이터 변환

직접 난 묶음을 대상으로 한 기존 기록은 다음 변환이 가능하다.

```text
WorkRecord 1건
→ WorkOperation 1건
→ WorkOperationTarget 1건
```

동·배드·구역 전체를 대상으로 한 과거 이력은 당시 포함된 난 묶음을 정확히 복원하기 어렵다.

따라서 다음처럼 처리한다.

```text
legacyScopeOnly = true
targetResolutionStatus = UNAVAILABLE
```

현재 위치를 기준으로 과거 대상을 임의 생성하지 않는다.

## 5.3 전환 완료 후

신규 기능이 안정화된 뒤에만 다음을 검토한다.

```text
기존 work_records 쓰기 API 제거
기존 화면 신규 API로 통합
기존 테이블 읽기 전용 전환
```

첫 배포에서 기존 테이블을 삭제하지 않는다.

## 단계 1. 화분 크기 정규화

자동 그룹 구현 전에 화분 크기 값부터 정리한다.

### 작업

* 현재 `pot_size` 값 목록 추출
* 표준 코드 정의
* 기존 값 매핑표 작성
* 매핑 불가능한 데이터 검수 목록 생성
* `pot_size_code` 추가
* 기존 표시 문자열은 스냅샷 또는 표시값으로 유지
* 신규 입력 UI에서 표준값만 선택하도록 변경

### 예시

```text
POT_2_5
POT_3
POT_3_5
POT_4
POT_5
HANGING
ETC
```

### 테스트

* 같은 규격의 다른 문자열이 하나의 코드로 매핑되는지 확인
* 미매핑 데이터가 자동으로 잘못 합쳐지지 않는지 확인

### 완료 조건

* 모든 활성 난 묶음에 표준 화분 크기 코드 존재
* 자유 문자열 신규 입력 차단

## 단계 2. 사용자 그룹 기반 기능

자동 그룹보다 먼저 저장형 사용자 그룹을 구현한다.

### 백엔드

* 사용자 그룹 CRUD
* 보관 처리
* 구성원 추가·제거
* 다중 구성원 추가
* 난 묶음 상세에서 소속 그룹 조회
* 중복 소속 방지
* 수량 0 또는 종료된 난 묶음 표시 정책 적용

### API

```http
GET    /api/orchid-group-collections
POST   /api/orchid-group-collections
GET    /api/orchid-group-collections/{id}
PATCH  /api/orchid-group-collections/{id}
POST   /api/orchid-group-collections/{id}/archive

POST   /api/orchid-group-collections/{id}/members
DELETE /api/orchid-group-collections/{id}/members/{orchidGroupId}

GET    /api/orchid-groups/{orchidGroupId}/collections
```

### 프론트엔드

* 사용자 그룹 목록
* 그룹 생성·수정·보관
* 난 묶음 다중 선택 후 그룹 추가
* 그룹 상세에서 구성원 조회·제거
* 총 묶음 수와 총수량 표시

### 완료 조건

* 한 난 묶음이 여러 사용자 그룹에 소속 가능
* 속성이나 위치가 바뀌어도 소속 유지
* 보관 그룹은 기본 목록에서 제외되지만 과거 작업에서는 조회 가능

## 단계 3. 자동 그룹 조회

### 백엔드

`OrchidGroup`을 다음 기준으로 집계한다.

```text
variety_id
age_year
pot_size_code
```

필터:

```text
active = true
quantity > 0
```

### API

```http
GET /api/orchid-groups/derived-groups
```

## 단계 6. 기간 작업과 대상별 진행 상태

### 상태 명령

```http
POST /api/work-operations/{id}/start
POST /api/work-operations/{id}/pause
POST /api/work-operations/{id}/resume
POST /api/work-operations/{id}/complete
POST /api/work-operations/{id}/cancel
```

대상별 명령:

```http
POST /api/work-operations/{id}/targets/{targetId}/start
POST /api/work-operations/{id}/targets/{targetId}/complete
POST /api/work-operations/{id}/targets/{targetId}/skip
```

대상 관리:

```http
POST /api/work-operations/{id}/targets
POST /api/work-operations/{id}/targets/refresh
DELETE /api/work-operations/{id}/targets/{targetId}
```

### 상태 규칙

* `DRAFT`: 자유롭게 수정 가능
* `PLANNED`: 대상 새로고침 가능
* `IN_PROGRESS`: 완료되지 않은 대상 추가 가능
* `PAUSED`: 도메인 효과 실행 금지
* `COMPLETED`: 대상 추가·제거 금지
* `CANCELED`: 완료 효과가 없는 경우에만 취소
* 구조 변경 완료 작업: 일반 취소 불가

### 작업 부담 최소화

매일 작업 일지를 작성하도록 강제하지 않는다.

```text
작업 시작일 지정
→ 처리한 난 묶음을 묶어서 완료
→ 며칠 뒤 나머지 난 묶음 완료
→ 전체 작업 종료
```

### 완료 조건

* 여러 날 이어지는 작업을 하나의 작업으로 유지
* 대상별 완료일이 다를 수 있음
* 전체 진행률 표시 가능
* 일부 대상만 건너뛰거나 제외 가능

## 단계 7. 작업 효과 처리 프레임워크

```java
public interface WorkEffectHandler {
```

### 멱등성

같은 완료 요청이 두 번 호출되어도 효과가 한 번만 적용되어야 한다.

```text
effect_applied_at
```

프론트엔드 재시도, 네트워크 지연, 중복 클릭을 모두 고려한다.

### 트랜잭션

대상 하나의 구조 변경은 한 트랜잭션으로 처리한다.


중간에 실패하면 전부 롤백한다.

### 완료 조건

* 작업 유형별 도메인 변경 로직이 분리됨
* 중복 요청으로 난 묶음이 중복 생성되지 않음
* 실패 시 부분 변경이 남지 않음

## 단계 8. 다중 난 묶음 생성

### 사용 사례

```text
초기 데이터 등록
동일 입고에서 여러 난 묶음 생성
여러 위치에 난 묶음 일괄 배치
분갈이 준비 결과 등록
```

### 처리 흐름

```text
다중 생성 작업 생성
→ 생성 행 입력
→ 위치와 속성 검증
→ 난 묶음 일괄 생성
→ 생성된 난 묶음을 작업 효과로 연결
```

### 생성 행

```text
품종
년생
화분 크기
수량
위치
시작 칸
끝 칸
배치 규격
메모
사용자 그룹 소속
```

### 데이터 관계

다중 생성은 원본 난 묶음 없이 결과 난 묶음만 가진다.

```text
WorkOperation
 └─ WorkEffect
     ├─ 생성 난 묶음 A
     ├─ 생성 난 묶음 B
     └─ 생성 난 묶음 C
```

### 검증

* 위치 중복
* 수량 0 이하
* 비활성 품종
* 잘못된 화분 크기
* 배드 범위 초과
* 동일 요청 중복 실행

### 취소 정책

아직 다른 작업, 판매, 이동, 입고와 연결되지 않은 생성 작업만 취소 가능하다.

연결 데이터가 존재하면 보정 작업을 사용한다.

### 완료 조건

* 여러 난 묶음을 하나의 작업 단위로 생성
* 각 난 묶음에서 생성 원인 작업 조회 가능
* 생성 작업 상세에서 모든 결과 난 묶음 조회 가능

## 단계 9. 분갈이 작업

분갈이는 가장 마지막에 구현한다.

### 대상별 입력

```text
원본 난 묶음
작업일
입력 수량

결과 난 묶음 목록
- 결과 수량
- 새 년생
- 새 화분 크기
- 위치
- 배치 규격
- 메모

손실 수량
손실 사유
사용자 그룹 상속
```

### 수량 검증

```text
입력 수량
=
결과 수량 합계
+
손실 수량
+
원본 잔여 수량
```

오차를 허용할 경우 반드시 수량 보정 사유를 기록한다.

### 전체 분갈이

```text
원본 수량 200
결과 수량 195
손실 수량 5
원본 잔여 수량 0

→ 원본 난 묶음 종료
→ 결과 난 묶음 생성
```

### 부분 분갈이

```text
원본 수량 200
작업 투입 수량 120
결과 수량 118
손실 수량 2
원본 잔여 수량 80

→ 원본 수량 80으로 감소
→ 결과 난 묶음 생성
→ 원본은 활성 유지
```

### 계보

```text
원본 A
 ├─ REPOTTED_TO → 결과 B
 └─ REPOTTED_TO → 결과 C
```

### 자동 그룹 처리

별도 로직 없이 결과 난 묶음의 새 속성을 기준으로 자동 그룹 조회 결과가 바뀐다.

### 사용자 그룹 처리

완료 전에 원본이 속한 사용자 그룹을 표시한다.

```text
우량주 관리             [상속]
2027년 봄 출하 후보      [미상속]
이번 주 분갈이 대상      [미상속]
```

### 완료 조건

* 전체·부분 분갈이 지원
* 결과 난 묶음 여러 개 생성 가능
* 수량 보존 검증
* 원본과 결과 계보 조회
* 작업 완료 중복 실행 차단
* 원본과 결과 양쪽에서 작업 이력 조회 가능

# 13. 첫 구현 범위

첫 번째 실제 구현은 다음 하나의 수직 흐름으로 제한한다.

```text
자동 그룹 또는 3동 전체 선택
→ 농약 작업 등록
→ 실제 난 묶음 대상 스냅샷 저장
→ 작업 완료
→ 각 난 묶음 상세 이력에서 작업 조회
→ 난 묶음을 다른 동으로 이동
→ 과거 농약 이력이 유지되는지 확인
```

이 흐름을 먼저 완성하면 다음 핵심 기반을 검증할 수 있다.

```text
그룹 대상 해석
상위 범위 전파
대상 스냅샷
작업 상태
하위 이력 조회
위치 변경 이후 이력 보존
```

분갈이와 다중 생성은 이 기반이 안정된 뒤에 추가한다.

# 15. 최종 완료 기준

다음 조건을 모두 만족하면 기능 구현이 완료된 것으로 본다.

* 자동 그룹은 현재 속성을 기준으로 정확히 계산된다.
* 사용자 그룹은 난 묶음 속성 변경과 무관하게 유지된다.
* 상위 범위 작업은 작업 당시 실제 난 묶음 목록을 보존한다.
* 하위 난 묶음 이력에서 전파된 작업을 확인할 수 있다.
* 난 묶음이 이동해도 과거 작업 이력이 바뀌지 않는다.
* 기간 작업에서 대상별 진행 상태를 관리할 수 있다.
* 다중 생성 결과가 하나의 작업과 연결된다.
* 분갈이 결과는 새로운 난 묶음과 계보로 보존된다.
* 구조 변경 작업이 중복 실행되지 않는다.
* 수량 변경 전후가 검증되고 기록된다.
* 완료된 구조 변경 작업은 삭제 대신 보정할 수 있다.
* 기존 작업 이력과 신규 작업 이력을 함께 조회할 수 있다.
* 기능 비활성화 시 기존 운영 화면으로 안전하게 복귀할 수 있다.
