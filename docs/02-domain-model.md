# 도메인 모델

## 1. 농장 구조

```text
House
 └─ PhysicalBed
    └─ BedZone
       └─ BedZoneSegment
          └─ OrchidGroup
```

### House

농장의 동이다.

- 번호와 이름을 가진다.
- 기본 운영 기준은 15개 동이다.
- 하나의 동은 여러 물리 배드를 가진다.

### PhysicalBed

동 내부의 실제 배드다.

- 기본 운영 기준은 동마다 3개다.
- 실제 농장 방향 기준으로 좌→우 표시 순서를 가진다.
- 길이, 폭, 철사 줄 수, 지지대 간격 같은 참고 정보를 가질 수 있다.

### BedZone

물리 배드를 실제 운영 단위로 나눈 논리 구역이다.

- 기본값은 각 배드의 좌/우 구역이다.
- `LEFT`, `RIGHT`, `CUSTOM`, `HANGING` 같은 구역 방향/유형을 가진다.
- 난 묶음 배치와 작업 이력의 기본 위치 단위다.

### BedZoneSegment

논리 구역을 더 세분화한 배치 단위다.

- 앞/중간/끝 구간을 구분할 수 있다.
- 자리 차지 계산과 구간 배치 검증에 사용한다.
- `여유`, `표준`, `확장`, `압축`, `임시` 같은 수용 모드를 가질 수 있다.

### OrchidGroup

난 묶음은 현재 시스템의 핵심 관리 단위다.

- 모든 난 묶음은 반드시 하나의 품종 마스터와 연결된다.
- 품종명, 속, 수량, 화분 크기, 연차, 상태, 메모를 가진다.
- 하나의 논리 구역 또는 세그먼트에 배치된다.
- 같은 구역 안에서 표시 순서를 가진다.
- 이동 시 위치 이동 작업 이력이 자동 생성된다.

### Variety

품종 기준 정보다.

- 속, 품종명, 별칭, 기본 화분 크기, 판매 사용 여부를 가진다.
- 연결된 난 묶음 수와 총 보유 수량 같은 요약값 조회 기준이 된다.
- 운영 데이터 보존을 위해 삭제보다 비활성화를 우선한다.

### InboundRecord

입고 기록이다.

- 입고일, 입고 유형, 품종, 수량, 임시 위치, 메모를 가진다.
- 유리병 모종은 포트 작업 전까지 임시보관 흐름을 가진다.
- 즉시 배치 입고는 실제 수량과 논리 구역 배치 정보를 함께 가진다.
- 난 묶음이 생성된 뒤에는 취소보다 상태 전이를 우선한다.

### Material

약제, 비료, 일반 자재 기준 정보다.

- 분류, 자재명, 제조사, 규격, 보유 수량, 보관 위치, 사용법을 가진다.
- 작업 기록과 운영 메모 입력 시 정규화 기준으로 사용한다.
- 운영 데이터 보존을 위해 삭제보다 비활성화를 우선한다.

## 2. 작업 도메인

```text
WorkType 1 ─ N WorkRecord
```

### WorkType

작업 유형이다.

기본 예시:

- 농약
- 비료
- 분갈이
- 상태 기록
- 일반 메모
- 잎 정리
- 잡초 정리
- 단화/꽃 정리
- 위치 이동

사용자 정의 작업 유형을 추가, 수정, 비활성화, 정렬할 수 있다.

### WorkRecord

실제 작업 이력이다.

- 작업일
- 작업 유형
- 대상 유형: 농장, 동, 물리 배드, 논리 구역, 난 묶음
- 작업자
- 약제명/비료명
- 희석 배수/농도
- 수량
- 메모
- 이동 전/후 위치

위치 이동은 삭제보다 이력 보존을 우선한다.

## 3. 거래처·판매 도메인

```text
BusinessPartner 1 ─ N SalesSlip 1 ─ N SalesSlipItem
```

### BusinessPartner

거래처다.

- 일반 도매상
- 소매상
- 경매장

경매장도 거래처로 관리하되, 정산 설정과 경매 lot 연결이 필요하다.

### SalesSlip

판매 전표다.

- 일반 판매와 경매 판매를 구분할 수 있다.
- 판매일, 거래처, 입금 상태, 판매 상태, 총액, 메모를 가진다.
- A5 출력 대상이다.

### SalesSlipItem

판매 품목이다.

- 품목명/품종명
- 속명
- 규격
- 수량
- 단가
- 금액
- 선택적으로 난 묶음과 연결 가능

전표 생성 시점에 바로 재고 차감하지 않는다. 재고 연동은 출고 완료 또는 판매 확정 기준으로 확장한다.

## 4. 경매 추적 도메인

```text
AuctionLot
 ├─ AuctionAttempt
 │   └─ AuctionResultLine
 └─ AuctionStatusHistory

AuctionSettlement
 └─ AuctionSettlementLine
```

### AuctionLot

경매장에 출하된 물건의 추적 단위다.

- 출하일
- 경매장
- 품목/품종/등급
- 출하 수량
- 판매 수량
- 대기 수량
- 반환 수량
- 현재 상태

### AuctionAttempt

경매 시도 이력이다.

- 같은 lot이 유찰 후 여러 번 경매될 수 있다.
- 경매일, 차수, 결과 상태를 기록한다.

### AuctionResultLine

경매 결과 원본 행이다.

- 같은 lot이 여러 단가로 나뉘어 판매될 수 있으므로 결과 행은 별도로 보존한다.

### AuctionStatusHistory

상태 변경 이력이다.

- 출하, 유찰, 재경매 대기, 판매, 반환, 수량 보정 등을 기록한다.
- 추정/수동확인 여부를 남긴다.

### AuctionSettlement

경매장 정산 묶음이다.

- 경매장과 정산 기간을 기준으로 생성한다.
- lot 또는 결과 행을 연결한다.
- 정산 금액, 입금 상태, 예상 입금일을 가진다.

## 5. 입금·잔액 도메인

```text
PartnerSettlementSettings
PartnerPaymentEvent
PartnerBalanceSummary
```

### PartnerSettlementSettings

거래처별 정산 규칙이다.

- 정산 단위
- 결제 지연일
- 결제일 모드
- 허용 오차
- 계좌/입금자명 매칭 설정
- 자동 정산/자동 매칭 사용 여부

### PartnerPaymentEvent

입금 이벤트다.

- 판매 전표 입금
- 경매 정산 입금
- 대상 미지정 입금
- 부분입금
- 초과입금

입금 처리는 삭제보다 이벤트 보존을 우선한다.
