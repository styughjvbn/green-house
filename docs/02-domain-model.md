# 도메인 모델

## 1. 농장 구조

```text
House
 └─ PhysicalBed
    └─ BedZone
       └─ BedZoneSegment
          └─ OrchidGroup
```

### House

농장의 동이다.

- 번호와 이름을 가진다.
- 기본 운영 기준은 15개 동이다.
- 하나의 동은 여러 물리 배드를 가진다.

### PhysicalBed

동 내부의 실제 배드다.

- 기본 운영 기준은 동마다 3개다.
- 실제 농장 방향 기준으로 좌→우 표시 순서를 가진다.
- 길이, 폭, 철사 줄 수, 지지대 간격 같은 참고 정보를 가질 수 있다.
- 실제 농장에서는 "다이" 고 칭한다.

### BedZone

물리 배드를 실제 운영 단위로 나눈 논리 구역이다.

- 기본값은 각 배드의 좌/우 구역이다.
- `LEFT`, `RIGHT`, `CUSTOM`, `HANGING` 같은 구역 방향/유형을 가진다.
- 난 묶음 배치와 작업 이력의 기본 위치 단위다.

### BedZoneSegment

논리 구역을 더 세분화한 배치 단위다.

- 앞/중간/끝 구간을 구분할 수 있다.
- 자리 차지 계산과 구간 배치 검증에 사용한다.
- `여유`, `표준`, `확장`, `압축`, `임시` 같은 수용 모드를 가질 수 있다.

### OrchidGroup

난 묶음은 현재 시스템의 핵심 관리 단위다.

- 모든 난 묶음은 반드시 하나의 품종 마스터와 연결된다.
- 품종명, 속, 수량, 예약 수량, 화분 크기, 초기 년생, 상태, 메모를 가진다.
- 하나의 논리 구역 또는 세그먼트에 배치된다.
- 같은 구역 안에서 표시 순서를 가진다.
- 이동 시 위치 이동 작업 이력이 자동 생성된다.
- 가용 수량은 `수량 - 예약 수량`으로 계산한다.
- 화분 크기는 표준 `potSizeCode`를 가지며 원본 표시 문자열과 분리한다.
- 의미가 불명확한 기존 화분 값은 임의 통합하지 않고 `UNMAPPED`로 검수한다.

### OrchidGroupCollection / OrchidGroupCollectionMember

사용자가 목적에 따라 난 묶음을 지속적으로 묶는 관리 그룹과 소속 이력이다.

- 하나의 난 묶음은 여러 사용자 그룹에 동시에 속할 수 있다.
- 위치, 년생, 화분 크기, 상태가 바뀌어도 소속은 자동으로 바뀌지 않는다.
- 구성원 해제는 소속 행을 삭제하지 않고 `removedAt`을 기록한다.
- 그룹은 삭제보다 `ARCHIVED` 보관을 우선하며, 보관 후 소속 변경을 막는다.
- 같은 그룹과 난 묶음의 활성 소속은 하나만 허용한다.

### DerivedOrchidGroup

품종, 현재 년생, 표준 화분 크기가 같은 활성 난 묶음을 조회 시점에 계산한 자동 그룹이다.

- 별도 엔티티나 동기화 데이터를 저장하지 않는다.
- 그룹 키는 `varietyId:ageYear:potSizeCode` 형식이며 현재 조회 조건을 나타낼 뿐 영구 ID가 아니다.
- 수량 0과 종료·폐기·판매 완료 난 묶음, `UNMAPPED` 화분 값은 제외한다.
- 난 묶음 속성이 변경되면 다음 조회부터 기존 그룹에서 빠지고 새 조건의 그룹에 포함된다.

### Variety

품종 기준 정보다.

- 속, 품종명, 별칭, 기본 화분 크기, 판매 사용 여부를 가진다.
- 속과 품종명 조합이 품종의 자연키 역할을 하며, 같은 조합은 중복 등록할 수 없다.
- 연결된 난 묶음 수와 총 보유 수량 같은 요약값 조회 기준이 된다.
- 운영 데이터 보존을 위해 삭제보다 비활성화를 우선한다.

### InboundRecord

입고 기록이다.

- 입고일, 입고 유형, 품종, 수량, 임시 위치, 메모를 가진다.
- 유리병 모종은 포트 작업 전까지 임시보관 흐름을 가진다.
- 즉시 배치 입고는 실제 수량과 논리 구역 배치 정보를 함께 가진다.
- 난 묶음이 생성된 뒤에는 취소보다 상태 전이를 우선한다.

### Material

약제, 비료, 일반 자재 기준 정보다.

- 분류, 자재명, 제조사, 규격, 보유 수량, 보관 위치, 사용법을 가진다.
- 작업 기록과 운영 메모 입력 시 정규화 기준으로 사용한다.
- 운영 데이터 보존을 위해 삭제보다 비활성화를 우선한다.

## 2. 작업 도메인

```text
WorkType 1 ─ N WorkRecord
WorkType 1 ─ N WorkOperation 1 ─ N WorkOperationTarget 1 ─ 1 WorkTargetExecution
                                      └─ N WorkAppliedEffect
```

### WorkType

작업 유형이다.

기본 예시:

- 농약
- 비료
- 분갈이
- 상태 기록
- 일반 메모
- 잎 정리
- 잡초 정리
- 단화/꽃 정리
- 위치 이동

사용자 정의 작업 유형을 추가, 수정, 비활성화, 정렬할 수 있다.

### WorkRecord

실제 작업 이력이다.

- 작업일
- 작업 유형
- 대상 유형: 농장, 동, 물리 배드, 논리 구역, 난 묶음
- 작업자
- 약제명/비료명
- 희석 배수/농도
- 수량
- 작업별 가변 상세 정보는 `details`에 보관한다.
- 메모
- 이동 전/후 위치
- 실수 입력은 삭제하지 않고 취소 상태와 취소 사유로 남긴다.

위치 이동은 삭제보다 이력 보존을 우선한다.

### WorkOperation

상위 범위와 그룹을 포함하는 신규 작업 전체 단위다.

- 원본 선택 범위와 실제 난 묶음 대상을 분리한다.
- 현재 대상 선택은 동 전체, 자동 그룹, 사용자 그룹, 직접 다중 선택을 지원한다.
- 저장 시 실제 대상과 당시 위치·속성·수량 스냅샷을 확정한다.
- 상위 범위 작업은 난 묶음별로 복제하지 않는다.
- 대상별 실행 상태를 별도로 가진다.
- 전체 작업은 계획·진행·일시중지·완료·취소 상태를 가진다.
- 각 대상은 대기·진행·완료·건너뜀·취소·실패 상태와 처리 시각·작업자·결과 상세를 가진다.
- 대상 완료는 작업 효과 handler를 거치며 효과 적용과 완료 상태 변경을 하나의 트랜잭션으로 처리한다.
- 같은 대상의 같은 효과는 한 번만 적용하고 `WorkAppliedEffect`에 handler, 입력·결과, 적용 시각을 보존한다.
- 원본 대상이 없는 다중 생성 작업은 작업 단위 효과에 생성된 난 묶음들을 `CREATED` 관계로 연결한다.
- 다중 생성 요청은 요청 식별자를 보존해 같은 요청 재시도 시 결과를 다시 생성하지 않는다.
- 연결되지 않은 다중 생성 결과의 취소는 난 묶음을 삭제하지 않고 수량 0·`생성 취소` 상태로 보존한다.
- 취소 시 사용자 그룹 소속을 해제하고 작업 효과에 취소 시각을 기록한다.
- 작업 효과로 생성되거나 연결된 난 묶음에서도 원인 작업을 통합 이력으로 조회한다.
- 분갈이는 원본을 잠근 뒤 투입 수량을 차감하고 결과 생성·사용자 그룹 상속·계보·작업 효과를 한 트랜잭션에서 기록한다.
- 전체 분갈이는 원본을 수량 0·`종료` 상태로 보존하고, 부분 분갈이는 차감 후 잔여 수량으로 계속 운영한다.

### OrchidGroupLineage

분갈이·분주·병합처럼 생육 단위가 바뀌는 구조 변경에서 원본과 결과 난 묶음을 연결한다.

- 원본과 결과는 서로 다른 난 묶음이어야 한다.
- 관계 유형과 원인 `WorkOperation`을 보존한다.
- 작업 당시 원본 수량과 결과 수량을 함께 보존한다.
- 난 묶음 기준으로 직접 원본과 직접 결과를 양방향 조회한다.

기존 `WorkRecord`는 과거 기록과 위치 이동 같은 기존 시스템 기록을 위해 유지한다. 난 묶음 이력 조회에서는 두 모델을 합쳐 표시한다.

## 3. 거래처·판매 도메인

```text
BusinessPartner 1 ─ N SalesSlip 1 ─ N SalesSlipItem 1 ─ N SalesSlipItemAllocation
```

### BusinessPartner

거래처다.

- 일반 도매상
- 소매상
- 경매장

경매장도 거래처로 관리하되, 정산 설정과 경매 lot 연결이 필요하다.

### SalesSlip

판매 전표다.

- 일반 판매와 경매 판매를 구분할 수 있다.
- 판매일, 거래처, 입금 상태, 판매 상태, 총액, 메모를 가진다.
- A5 출력 대상이다.

### SalesSlipItem

판매 품목이다.

- 품목명/품종명
- 속명
- 규격
- 수량
- 단가
- 금액
- 하나 이상의 난 묶음 배분과 연결된다.

### SalesSlipItemAllocation

판매 품목과 난 묶음 사이의 배분 연결이다.

- 하나의 판매 품목은 여러 난 묶음에 나눠 배분할 수 있다.
- 배분 수량 합계는 품목 수량과 같아야 한다.
- 판매 품목의 품종명과 난 묶음 품종명이 일치해야 한다.
- 전표 저장 시 배분 수량만큼 난 묶음 `reservedQuantity`를 증가시킨다.

### SalesInventoryMovement

판매 전표로 인한 예약/해제/출고 이력이다.

- `SALES_RESERVE`
- `SALES_RELEASE`
- `SALES_OUTBOUND`

전표 저장 시점에 바로 재고 차감하지 않는다. 전표 저장은 예약만 반영하고, 실제 수량 차감은 출고 완료 시점에 처리한다.

## 4. 경매 추적 도메인

```text
AuctionLot
 ├─ AuctionAttempt
 │   └─ AuctionResultLine
 └─ AuctionStatusHistory

AuctionSettlement
 └─ AuctionSettlementLine
```

### AuctionLot

경매장에 출하된 물건의 추적 단위다.

- 출하일
- 경매장
- 품목/품종/등급
- 출하 수량
- 판매 수량
- 대기 수량
- 반환 수량
- 현재 상태

### AuctionAttempt

경매 시도 이력이다.

- 같은 lot이 유찰 후 여러 번 경매될 수 있다.
- 경매일, 차수, 결과 상태를 기록한다.

### AuctionResultLine

경매 결과 원본 행이다.

- 같은 lot이 여러 단가로 나뉘어 판매될 수 있으므로 결과 행은 별도로 보존한다.

### AuctionStatusHistory

상태 변경 이력이다.

- 출하, 유찰, 재경매 대기, 판매, 반환, 수량 보정 등을 기록한다.
- 추정/수동확인 여부를 남긴다.

### AuctionSettlement

경매장 정산 묶음이다.

- 경매장과 정산 기간을 기준으로 생성한다.
- lot 또는 결과 행을 연결한다.
- 정산 금액, 입금 상태, 예상 입금일을 가진다.

## 5. 입금·잔액 도메인

```text
PartnerSettlementSettings
PartnerPaymentEvent
PartnerBalanceSummary
```

### PartnerSettlementSettings

거래처별 정산 규칙이다.

- 정산 단위
- 결제 지연일
- 결제일 모드
- 허용 오차
- 계좌/입금자명 매칭 설정
- 자동 정산/자동 매칭 사용 여부

### PartnerPaymentEvent

입금 이벤트다.

- 판매 전표 입금
- 경매 정산 입금
- 대상 미지정 입금
- 부분입금
- 초과입금

입금 처리는 삭제보다 이벤트 보존을 우선한다.
