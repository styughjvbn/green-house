# v2 변경사항 요약

## 1. 변경 배경

기존 문서는 농장을 `15개 동 × 각 동 3개 배드` 구조로 정의했다.

그러나 실제 농장에서는 각 물리 배드가 중앙 관수 라인과 세로 철사 구조를 기준으로 좌/우 논리 구역처럼 활용된다. 또한 철사 사이에 분화를 걸거나 20구/15구 판을 끼워 분화를 담는 방식, 행잉 분화 배치 방식이 존재한다.

따라서 v2에서는 실제 운영 구조를 반영하여 `동 → 물리 배드 → 논리 구역 → 난 묶음` 구조로 문서를 수정했다.

---

## 2. 농장 구조 변경

### 기존

```text
농장
 └─ 동
    └─ 배드
       └─ 난 묶음
```

### 변경

```text
농장
 └─ 동
    └─ 물리 배드
       └─ 논리 구역
          └─ 난 묶음
```

### 핵심 변경

- 1동에는 3개의 물리 배드가 존재한다.
- 각 물리 배드는 좌/우 2개의 기본 논리 구역으로 활용된다.
- MVP 기준 1동에는 6개의 기본 논리 구역이 존재한다.
- 전체 농장은 15동 × 3물리 배드 × 2논리 구역 = 90개 기본 논리 구역을 가진다.

---

## 3. 농장맵 방향 변경

농장맵은 단순 조회 화면이 아니라 난 묶음을 생성하고 이동하는 작업 화면으로 확장한다.

MVP 포함:

- 논리 구역 클릭
- 구역별 난 묶음 조회
- 구역에서 난 묶음 추가
- 난 묶음 블록 UI
- 배치 수정 모드
- 난 묶음 전체 이동

MVP 제외:

- 자유 영역 드래그 생성
- 논리 구역 병합/분할
- 철사 단위 정밀 위치 관리
- 15구/20구 판 단위 정밀 배치
- 개별 화분 위치 지정

---

## 4. 난 묶음 이동 변경

난 묶음 이동은 단순 위치 변경이 아니라 작업 이력으로 기록한다.

MVP 규칙:

- 난 묶음 블록을 드래그 앤 드롭으로 이동한다.
- 배치 수정 모드에서만 이동 가능하다.
- 이동 전 확인 창을 표시한다.
- 이동은 전체 이동만 지원한다.
- 이동 완료 시 위치 이동 작업 이력을 자동 생성한다.

MVP 이후:

- 일부 수량 이동
- 난 묶음 자동 분할
- 일괄 이동

---

## 5. 작업 유형 변경

기존 MVP 작업 유형:

```text
농약
비료
분갈이
상태 기록
일반 메모
```

v2 추가 작업 유형:

```text
잎 정리
잡초 정리
단화 / 꽃 정리
위치 이동
```

현재 구현:

- 사용자가 설정 화면에서 작업 유형을 추가/수정/비활성화/정렬할 수 있다.
- 작업 유형은 템플릿(`PESTICIDE`, `FERTILIZER`, `REPOT`, `CLEANUP`, `STATUS`, `MEMO`, `MOVEMENT`)을 선택한다.
- `위치 이동`은 시스템 작업 유형으로 유지한다.

---

## 6. 판매 전표 변경

MVP에서는 기존과 같이 전표 생성과 출력 기능을 우선한다.

v2에서는 향후 확장을 위해 판매 품목이 난 묶음과 연결될 수 있도록 `orchid_group_id` nullable 구조를 추가했다.

MVP 이후 방향:

- 판매 품목과 난 묶음 연결
- 판매 확정 또는 출고 완료 시 난 묶음 수량 차감
- 전표 취소 또는 출고 취소 시 수량 복구
- 재고 변동 이력 관리

---

## 7. 출력물 변경

기존 문서에서는 A4 출력을 기준으로 했다.

v2에서는 출력 기본 규격을 A5로 변경한다.

원칙:

- 판매 전표는 A5 1장 기준
- 논리 구역/배드 관리표는 A5 여러 장 허용
- 글자를 작게 만들지 않음
- 한 장에 너무 많은 정보를 넣지 않음
- A4 출력은 필요 시 MVP 이후 옵션으로 제공

## 8. DB 변경 요약

기존 주요 테이블:

```text
houses
beds
orchid_groups
work_records
customers
sales_slips
sales_slip_items
```

v2 주요 테이블:

```text
houses
physical_beds
bed_zones
orchid_groups
work_types
work_records
customers
sales_slips
sales_slip_items
```

핵심 변경:

- `beds`를 `physical_beds`로 명확화
- `bed_zones` 추가
- `orchid_groups.bed_id` 대신 `orchid_groups.bed_zone_id` 사용
- `work_records`에 위치 이동 관련 필드 추가
- `sales_slip_items`에 `orchid_group_id nullable` 추가

## 10. 구현상 가장 중요한 판단

MVP에서 자유도 높은 정밀 지도 기능을 만들지 않는다.

대신 다음 원칙을 따른다.

```text
실제 물리 구조는 데이터 모델에 반영한다.
사용자 조작은 단순한 관리용 그리드로 제공한다.
난 묶음 생성은 구역 클릭 기반으로 제공한다.
난 묶음 이동은 블록 드래그 앤 드롭으로 제공한다.
위치 변경은 작업 이력으로 남긴다.
```

## 11. v2.1 추가 변경사항

### 11.1 물리 배드 방향 명확화

한 동 내부의 물리 배드는 실제 농장 구조와 동일하게 왼쪽에서 오른쪽으로 1배드, 2배드, 3배드 순서로 배치한다.

기존 문서 일부 예시는 물리 배드가 위에서 아래로 나열되는 것처럼 보일 수 있었으므로, v2.1에서는 동 상세 보기의 기본 방향을 좌→우로 명확히 수정했다.

### 11.2 전체 동 보기와 동 상세 보기 역할 분리

전체 동 보기는 제거하지 않는다. 다만 역할을 다음처럼 제한한다.

```text
전체 동 보기 = 15개 동 상태 확인 + 동 선택 내비게이션
동 상세 보기 = 난 묶음 생성, 이동, 정렬, 작업 기록을 수행하는 작업 화면
```

전체 동 보기에서는 난 묶음까지 세밀하게 표시하지 않고, 동별 상태 이상, 분갈이 예정, 최근 작업 상태 등을 요약 표시한다.

### 11.3 동 상세 보기의 가시성 보완

동 상세 보기에서는 물리 배드를 실제 방향대로 좌→우 배치하는 것을 기본으로 한다.
다만 가로 공간이 부족할 수 있으므로 다음 보기 옵션을 둘 수 있다.

```text
실제 방향 보기
회전 보기
배드별 보기
```

회전 보기는 화면 렌더링 방식일 뿐이며, 데이터 기준 방향은 항상 실제 농장 방향을 따른다.

### 11.4 난 묶음 이동 UX 분리

난 묶음 이동은 이동 범위에 따라 UX를 분리한다.

```text
같은 동 내부 이동 = 드래그 앤 드롭
다른 동으로 이동 = 목적 동/물리 배드/논리 구역 선택 방식
```

동 간 이동을 드래그만으로 처리하면 화면 전환, 스크롤, 오동작 문제가 커지므로 MVP에서는 목적지 선택 방식을 기본으로 한다.

### 11.5 DB 표시 순서 보강

`physical_beds`에 `display_order`를 추가하여 한 동 내부의 물리 배드 좌→우 표시 순서를 명시한다.

# v2.2 변경사항

## 13. 농장 현황과 난 묶음 관리 화면 역할 분리

v2.2에서는 농장맵을 하나의 화면에서 모든 기능을 처리하는 구조로 보지 않는다. 화면 목적에 따라 다음처럼 분리한다.

```text
농장 현황 = 전체 농장맵 기반 통합 조회 화면
난 묶음 관리 = 선택한 동의 상세맵 기반 작업 화면
```

### 농장 현황 화면

농장 현황 화면은 전체 농장 상태를 확인하고, 선택 범위의 난 묶음 목록을 조회하는 데 사용한다.

```text
전체 농장 보기
→ 동 클릭
→ 해당 동의 모든 난 묶음 목록 표시

줌인
→ 물리 배드 클릭
→ 해당 배드의 모든 난 묶음 목록 표시

더 줌인
→ 논리 구역 클릭
→ 해당 구역의 난 묶음 목록과 최근 작업 요약 표시
```

농장 현황 화면에서는 다음 작업을 직접 수행하지 않는다.

```text
난 묶음 생성
난 묶음 수정
난 묶음 삭제
난 묶음 드래그 이동
```

### 난 묶음 관리 화면

난 묶음 관리 화면은 선택한 동의 상세맵을 기반으로 난 묶음을 실제로 관리하는 화면이다.

```text
동 선택
→ 동 상세맵 표시
→ 논리 구역 선택
→ 난 묶음 추가, 수정, 삭제 또는 이동
```

## 14. 농장 현황 조회 API 추가

농장 현황 화면을 위해 다음 API를 추가한다.

```http
GET /api/farm-status/map
GET /api/farm-status/orchid-groups?targetType=HOUSE&targetId=1
GET /api/farm-status/orchid-groups?targetType=PHYSICAL_BED&targetId=3
GET /api/farm-status/orchid-groups?targetType=BED_ZONE&targetId=15
GET /api/farm-status/zoom?level=HOUSE&houseId=1
```

기존 `/api/dashboard/farm-map` 성격의 API는 대시보드 요약에 가깝게 축소하고, 농장 현황의 상세 조회는 `/api/farm-status/*`로 분리한다.

## 15. 개발 계획 변경

기존에는 농장맵 UI 구현과 난 묶음 관리 구현이 하나의 흐름으로 묶여 있었다. v2.2에서는 다음처럼 작업 단위를 나눈다.

```text
3단계: 대시보드와 농장 현황 조회 화면
4단계: 난 묶음 관리 상세맵
5단계: 난 묶음 CRUD
6단계: 맵 기반 난 묶음 이동
```

이렇게 분리하면 전체 조회 화면과 데이터 변경 화면의 복잡도를 낮출 수 있다.
