# 판매 관리 출하·경매 추적 확장 설계서

## 1. 문서 목적

현재 판매 관리 기능은 거래처와 판매 전표를 등록하고, 판매 내역을 조회하며, 판매 전표를 A5로 출력하는 기능을 중심으로 구현되어 있다.

그러나 `2026 내수 출하` 데이터는 단순 판매 전표 데이터가 아니라 다음 흐름을 포함한다.

```text
농장 출하
→ 경매장 도착
→ 경매 진행
→ 판매 또는 유찰
→ 유찰 물량 재경매
→ 최종 판매 또는 농장 반환
```

따라서 현재 판매 전표 중심 구조만으로는 다음 질문에 답하기 어렵다.

- 어떤 물건이 경매장에 출하되었는가?
- 어떤 물건이 판매되었는가?
- 어떤 물건이 유찰되어 다음 경매를 기다리고 있는가?
- 어떤 물건이 결국 농장으로 반환되었는가?
- 품종별, 등급별, 경매장별 유찰률과 평균 단가는 어떤가?
- 출하 수량과 최종 판매·반환·대기 수량이 맞는가?

이 문서는 기존 판매 관리 기능을 유지하면서, 첨부된 출하·경매 데이터를 안정적으로 이식하고 추적하기 위한 기능 수정 방향을 정리한다.

---

## 2. 현재 판매 관리 기능의 한계

현재 판매 관리 기능은 다음 구조를 기준으로 한다.

```text
Customer 1 ─ N SalesSlip 1 ─ N SalesSlipItem
```

이 구조는 일반 직접 판매 전표에는 적합하다.

```text
거래처 선택
→ 판매일 입력
→ 품목·수량·단가 입력
→ 총액 계산
→ 전표 출력
```

하지만 경매 출하 데이터에는 다음 특성이 있다.

1. 판매일과 출하일이 다를 수 있다.
2. 같은 물건이 여러 번 경매에 등장할 수 있다.
3. 유찰된 물건은 다음 경매에 다시 잡힐 수 있다.
4. 최종적으로 판매되지 않고 농장으로 돌아올 수 있다.
5. 경매장에 있는 물량은 농장 재고도 아니고 판매 완료 물량도 아니다.
6. 같은 출하일·품종·경매장이라도 등급 또는 수량이 여러 행으로 나뉠 수 있다.

따라서 경매 데이터는 `SalesSlipItem`에 바로 넣기보다 별도 추적 도메인으로 분리해야 한다.

## 3. 첨부 데이터 분석 요약

첨부 CSV 기준 주요 데이터 분포는 다음과 같다.

| 항목 | 값 |
|---|---:|
| 전체 행 수 | 1,732행 |
| 출하 행 | 576행 |
| 경매 행 | 1,156행 |
| 데이터 기간 | 2025-05-04 ~ 2026-05-04 |
| 출하 분수량 합계 | 38,973분 |
| 출하 상자 합계 | 3,991상자 |
| 경매 행 기준 분수량 합계 | 45,990분 |
| 경매 매출 합계 | 289,548,360원 |
| 금액이 있는 경매 판매 수량 | 37,386분 |
| 유찰 행 기준 수량 | 8,604분 |
| 반환추정 행 | 16행, 596분 |

주의할 점은 `경매 행 기준 분수량 합계`를 실제 판매 또는 출하 물량으로 보면 안 된다는 것이다.

유찰된 물건은 다음 경매에 다시 등장하므로, 경매 행의 분수량을 단순 합산하면 같은 물건이 중복 집계된다.

예시:

```text
100분 출하
→ 1차 경매 유찰 100분
→ 2차 경매 판매 100분
```

이 경우 경매 행의 분수량 합계는 200분처럼 보이지만, 실제 출하 물량은 100분이다.

따라서 핵심은 행 합계가 아니라 `출하 묶음별 상태 추적`이다.

## 4. 추적 기준 키 검토

사용자가 제안한 기준 키는 다음과 같다.

```text
출하일자 + 품종명 + 등급 + 경매장
```

이 키는 업무상 조회 기준으로는 적절하다.

다만 DB의 고유키 또는 자동 매칭의 절대 기준으로 쓰기에는 부족하다.

### 4.1 중복 문제

첨부 데이터 기준 중복 현황은 다음과 같다.

| 기준 | 결과 |
|---|---:|
| 출하 행에서 `[출하일자, 품종명, 등급, 경매장]` 중복 그룹 | 47개 |
| 중복 그룹에 포함된 출하 행 | 95행 |
| 경매 행에서 `[출하일자, 품종명, 등급, 경매장]` 중복 그룹 | 232개 |
| 중복 그룹에 포함된 경매 행 | 840행 |

즉, 해당 조합은 `업무상 묶음 기준`으로는 좋지만 `유일 식별자`는 아니다.

### 4.2 등급 누락 문제

첨부 데이터에서 등급 누락이 많다.

| 구분 | 등급 누락 |
|---|---:|
| 출하 행 | 208 / 576행 |
| 경매 행 | 237 / 1,156행 |

또한 출하 시점에는 등급이 비어 있고 경매 시점에 등급이 붙는 경우가 많다.

정확 키 기준으로 출하 행과 경매 행을 비교하면 매칭되지 않는 출하 키가 108개였지만, 등급을 제외한 다음 기준을 사용하면 대부분 매칭 가능했다.

```text
출하일자 + 품종명 + 경매장
```

따라서 매칭 키는 다음처럼 단계화한다.

| 단계 | 기준 | 처리 |
|---|---|---|
| 1차 정확 매칭 | 출하일자 + 품종명 + 등급 + 경매장 | 자동 확정 가능 |
| 2차 보정 매칭 | 출하일자 + 품종명 + 경매장 | 확인 필요 또는 중간 신뢰도 |
| 3차 수량 기반 매칭 | 출하수량, 경매수량, 유찰수량, 반환수량 비교 | 수동 검수 권장 |

### 4.3 권장 키 구조

시스템 내부에서는 별도 ID를 사용한다.

```text
시스템 고유키: auction_shipment_lot.id
업무 조회 키: 출하일자 + 품종명 + 등급 + 경매장
보조 매칭 키: 출하일자 + 품종명 + 경매장
```

## 5. 권장 도메인 구조

기존 판매 전표 도메인은 유지한다.

```text
Customer
SalesSlip
SalesSlipItem
```

여기에 경매 추적 도메인을 추가한다.

```text
AuctionShipment
AuctionShipmentLot
AuctionAttempt
AuctionResultLine
AuctionLotStatusHistory
ImportBatch
ImportRow
```

역할은 다음과 같다.

```text
SalesSlip = 거래 증빙과 출력 중심
AuctionShipmentLot = 출하된 물건 추적 중심
AuctionAttempt = 경매 시도 이력
AuctionResultLine = 실제 경매 결과 행
InventoryMovement = 수량 변화 근거
```

```text
2025-05-04 / 음성 경매장 출하
```

## 7. 상태 모델

경매 출하 추적에는 판매 전표 상태와 별도 상태가 필요하다.

권장 상태는 다음과 같다.

| 상태 | 의미 |
|---|---|
| 출하완료 | 농장에서 경매장으로 출하됨 |
| 경매대기 | 아직 경매 결과가 없음 |
| 경매진행 | 경매 결과가 일부 존재함 |
| 판매완료 | 출하 수량이 모두 판매됨 |
| 부분판매 | 일부 수량만 판매됨 |
| 유찰 | 해당 경매에서 판매되지 않음 |
| 재경매대기 | 유찰 후 다음 경매 대기 중 |
| 반환추정 | 데이터상 반환으로 추정됨 |
| 반환완료 | 사용자가 반환을 확인함 |
| 수량불일치 | 출하·판매·유찰·반환 수량이 맞지 않음 |
| 확인필요 | 자동 판단이 어렵거나 원본 검수가 필요함 |
| 취소 | 잘못 등록되어 취소됨 |

상태 전이 예시는 다음과 같다.

```text
출하 등록
→ 출하완료
→ 경매대기

경매 결과 존재 + 금액 > 0
→ 판매완료 또는 부분판매

경매 결과 존재 + 비고=유찰 또는 금액=0
→ 유찰
→ 재경매대기

유찰 후 다음 경매에서 판매
→ 부분판매 또는 판매완료

유찰 후 반환 데이터 발생
→ 반환추정
→ 사용자가 확인
→ 반환완료

수량 합계 불일치 또는 매칭 실패
→ 확인필요
```

## 8. 재고 처리 원칙

경매 출하는 일반 판매와 다르다.

일반 판매는 다음 흐름이다.

```text
판매 전표 작성
→ 판매 확정
→ 출고 완료
→ 농장 재고 차감
```

경매 출하는 다음 흐름이다.

```text
농장에서 경매장으로 출하
→ 농장에는 없음
→ 아직 판매된 것은 아님
→ 경매장 대기
→ 판매되면 매출 확정
→ 유찰되면 경매장 대기
→ 반환되면 농장 재고 복귀
```

따라서 경매장 출하 물량은 별도 상태로 봐야 한다.

```text
농장 보유 수량
경매장 위탁 수량
판매 완료 수량
반환 수량
```

권장 재고 이동 유형은 다음과 같다.

| 이동 유형 | 설명 |
|---|---|
| OUT_TO_AUCTION | 농장에서 경매장으로 출하 |
| AUCTION_SOLD | 경매장에서 판매 완료 |
| AUCTION_FAILED | 유찰 기록 |
| RETURN_FROM_AUCTION | 경매장에서 농장으로 반환 |
| ADJUSTMENT | 수량 보정 |
| CANCEL | 취소 |

수량에 영향을 주는 작업은 반드시 이력으로 남겨야 한다.

## 9. 기존 판매 전표와의 관계

경매 추적 데이터를 처음부터 `SalesSlip`으로 모두 변환하지 않는다.

이유는 다음과 같다.

1. 유찰 행은 판매 전표가 아니다.
2. 재경매 행은 같은 물건의 후속 상태다.
3. 반환 행은 매출이 아니라 재고 복귀다.
4. 경매장 정산은 일반 거래처 전표와 성격이 다르다.

권장 구조는 다음과 같다.

```text
판매 관리
├─ 판매 전표
│  ├─ 직접 판매
│  └─ A5 출력
│
├─ 출하·경매 추적
│  ├─ 출하 lot 목록
│  ├─ 경매 시도 타임라인
│  ├─ 유찰·재경매·반환 상태 추적
│  └─ 확인 필요 큐
│
├─ 경매장 정산
│  ├─ 경매장별 매출
│  ├─ 품종별 판매량과 평균 단가
│  └─ 월 정산표
│
└─ 거래처·경매장 관리
```

판매 완료된 경매 결과만 매출 집계에 포함한다.

필요하면 월말 정산 단계에서 경매장별 정산표를 생성하고, 이를 별도 `AuctionSettlement` 또는 `SalesSlip`과 연결한다.

현재 구조:

### 10.3 상세 패널

상세는 타임라인 형태가 적합하다.

예시:


### 10.4 확인 필요 큐

별도 확인 목록이 필요하다.

대상:

```text
출하 행은 있는데 경매 행이 없음
경매 행은 있는데 출하 행과 매칭 실패
등급이 출하와 경매에서 다름
출하 수량과 판매·반환·대기 수량이 맞지 않음
반환추정인데 사용자가 아직 확인하지 않음
유찰 후 일정 기간 이상 후속 경매 없음
검수 상태가 확인필 또는 전표누락분인 행
```

이 목록은 실제 운영에서 가장 중요하다.

사용자는 전체 데이터를 다 보지 않고, `문제 있는 행`만 먼저 처리할 수 있어야 한다.

## 11. 가져오기 기능 설계

단순 CSV 업로드 후 즉시 저장하는 방식은 위험하다.

권장 흐름은 다음과 같다.

```text
CSV 업로드
→ 컬럼 매핑
→ 날짜 정규화
→ 숫자 정규화
→ 품종명·등급·경매장 정규화
→ 출하 행과 경매 행 분리
→ 자동 매칭
→ 매칭 실패·애매한 행 표시
→ 사용자 검수
→ 최종 저장
```

### 11.1 정규화 규칙

날짜:

```text
2025-05-04
2025. 5. 4
2025.05.04
```

모두 ISO 날짜로 정규화한다.

```text
2025-05-04
```

등급:

```text
2-3
4-
4-5
대
소
특
```

등급은 숫자나 날짜로 변환하지 않고 문자열로 유지한다.

숫자:

```text
상자
분수량
단가
금액
```

콤마를 제거한 뒤 숫자로 변환한다.

단, 원본 문자열은 `ImportRow.rawDataJson`에 보존한다.

### 11.2 검수 상태

가져오기 결과는 다음 상태를 가질 수 있다.

```text
정상
자동매칭
보정매칭
수동확인필요
매칭불가
수량불일치
반환추정
원본오류
```

## 16. 최종 설계 판단

현재 판매 관리는 `전표 작성과 출력`에는 적합하지만, 첨부된 출하·경매 데이터는 `상태 추적과 이력 관리`가 핵심이다.

따라서 다음 방향이 가장 안전하다.

```text
1. 기존 SalesSlip은 유지한다.
2. 출하·경매 추적 도메인을 별도로 추가한다.
3. 출하 lot을 중심으로 경매 시도와 결과 행을 연결한다.
4. 출하일자 + 품종명 + 등급 + 경매장은 조회 키로 사용한다.
5. 내부 고유키는 별도 ID를 사용한다.
6. 등급 누락을 고려해 보조 매칭 키를 둔다.
7. 유찰·재경매·반환은 상태 이력으로 관리한다.
8. 재고 차감은 판매 전표 생성 시점이 아니라 출하·판매·반환 이벤트 기준으로 처리한다.
9. 확인 필요 큐를 만들어 운영자가 문제 있는 행만 검수할 수 있게 한다.
10. 원본 행은 반드시 보존한다.
```

이 구조를 적용하면 단순히 `얼마를 팔았는지`뿐 아니라 다음 질문까지 시스템에서 추적할 수 있다.

```text
어떤 물건이 아직 경매장에 남아 있는가?
어떤 물건이 유찰 후 재경매 중인가?
어떤 물건이 반환되어야 하는가?
어떤 품종이 유찰률이 높은가?
어떤 경매장이 평균 단가가 좋은가?
출하 수량과 최종 처리 수량이 맞는가?
```
