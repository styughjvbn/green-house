# 판매 관리 통합 문서

## 1. 목적

판매 관리는 거래처와 판매 전표를 등록하고, 판매 내역을 조회하며, 전표를 A5로 출력하는 기능이다.

현재 구현 범위:

- 거래처 등록·조회
- 판매 전표 등록·목록·상세 조회
- 기간·거래처·상태·키워드 필터
- 품목별 금액과 전표 합계 자동 계산
- 판매 전표 A5 미리보기·인쇄

판매 수량에 따른 난 묶음 자동 차감은 MVP 범위가 아니다.

---

## 2. 사용자 흐름

### 판매 전표 조회

```text
판매 관리 진입
→ 기간·거래처·판매 상태·입금 상태·키워드 필터
→ 판매 전표 선택
→ 우측 전표 상세 확인
→ 필요 시 A5 인쇄 미리보기
```

### 판매 전표 등록

```text
새 판매 전표 등록
→ 판매일·거래처 선택
→ 입금·판매 상태 입력
→ 판매 품목 추가
→ 수량·단가 입력
→ 품목 금액·총액 자동 계산
→ 저장
```

### 거래처 등록

```text
거래처 관리 탭
→ 거래처명·대표자·전화번호·주소·메모 입력
→ 저장
→ 신규 거래처가 전표 입력 대상으로 자동 선택
```

## 3. 화면 구성

경로: `/sales`

### 판매 전표 목록 탭

- 상단: 전표 목록/거래처 관리 탭, 엑셀 다운로드, 신규 전표 버튼
- 필터: 최근 30일, 거래처, 판매 상태, 입금 상태, 키워드
- 좌측: 판매 전표 목록
- 우측: 선택 전표 상세
- 화면이 좁으면 목록과 상세를 세로 배치
- 표 너비가 부족하면 표 내부 가로 스크롤

필터는 입력 즉시 클라이언트에서 적용된다. 화면의 `검색` 버튼은 현재 별도 동작이 없다.

### 전표 상세

- 전표번호, 판매일, 입금 상태, 판매 상태
- 거래처명, 대표자, 연락처, 주소, 거래처 메모
- 결제 방식, 전표 메모
- 품목명, 속명, 규격, 수량, 단가, 금액, 품목 메모
- 공급가액, 부가세, 총액

공급가액과 부가세는 화면에서 총액을 1.1로 나누어 표시하는 참고값이다. DB에는 총액만 저장한다.

### 거래처 관리 탭

- 좌측: 거래처 등록 폼
- 우측: 거래처 목록
- 거래처 선택 시 신규 전표의 거래처 선택값 변경

### A5 출력

경로: `/print/sales-slips/{salesSlipId}`

- A5 세로 1장 기준
- 거래처, 품목, 합계, 상태, 서명 영역 표시
- 브라우저 인쇄 시 메뉴와 화면용 버튼 숨김
- 품목이 많으면 한 장을 넘을 수 있으며 별도 페이지 분할 규칙은 아직 없음

## 4. 데이터 모델

### Customer

| 필드 | 설명 |
|---|---|
| id | 거래처 ID |
| name | 거래처명, 필수 |
| ownerName | 대표자명 |
| phone | 전화번호 |
| address | 주소 |
| memo | 메모 |

### SalesSlip

| 필드 | 설명 |
|---|---|
| id | 판매 전표 ID |
| slipNumber | 전표번호, 유일값 |
| saleDate | 판매일, 필수 |
| customer | 거래처, 필수 |
| totalAmount | 품목 금액 합계 |
| paymentStatus | 입금 상태 |
| salesStatus | 판매 상태 |
| paymentMethod | 결제 방법 |
| memo | 전표 메모 |
| items | 판매 품목 목록, 1개 이상 |

### SalesSlipItem

| 필드 | 설명 |
|---|---|
| id | 판매 품목 ID |
| orchidGroupId | 연결 난 묶음 ID, 선택 |
| itemName | 품목명, 필수 |
| genus | 속명 |
| spec | 규격 |
| quantity | 수량, 1 이상 |
| unitPrice | 단가, 0 이상 |
| amount | `quantity × unitPrice` |
| memo | 품목 메모 |

관계:

```text
Customer 1 ─ N SalesSlip 1 ─ N SalesSlipItem
OrchidGroup 1 ─ N SalesSlipItem (선택 연결)
```

---

## 5. 계산·번호 규칙

- 품목 금액: `수량 × 단가`
- 전표 총액: 모든 품목 금액의 합
- 전표번호: `S + YYYYMMDD + 일자별 3자리 순번`
- 예시: `S20260703-001`
- 입금 상태 기본값: `미입금`
- 판매 상태 기본값: `작성중`

현재 UI 상태값:

- 입금 상태: `미입금`, `입금 완료`
- 판매 상태: `작성중`, `출고 완료`
- 목록 필터에는 `판매 확정`도 있으나 신규 전표 폼에서는 선택 불가

## 6. API

공통 응답:

```json
{
  "data": {},
  "message": null
}
```

### 거래처

| Method | Path | 설명 |
|---|---|---|
| GET | `/api/customers?keyword=...` | 거래처 목록·이름 검색 |
| POST | `/api/customers` | 거래처 등록 |

### 판매 전표

| Method | Path | 설명 |
|---|---|---|
| GET | `/api/sales-slips?customerId=&from=&to=` | 전표 목록 조회 |
| GET | `/api/sales-slips/{salesSlipId}` | 전표 상세 조회 |
| POST | `/api/sales-slips` | 전표 등록 |
| GET | `/api/sales-slips/{salesSlipId}/print` | A5 출력 데이터 조회 |

전표 등록 예시:

```json
{
  "saleDate": "2026-07-03",
  "customerId": 1,
  "paymentStatus": "미입금",
  "salesStatus": "작성중",
  "paymentMethod": "계좌이체",
  "memo": "7월 출하",
  "items": [
    {
      "orchidGroupId": null,
      "itemName": "카틀레야 A",
      "genus": "카틀레야",
      "spec": "4치",
      "quantity": 10,
      "unitPrice": 25000,
      "memo": null
    }
  ]
}
```

서버가 품목 금액과 전표 총액을 다시 계산한다. 클라이언트 계산값은 요청하지 않는다.

## 7. 검증·오류

- 거래처명 필수
- 판매일·거래처 필수
- 전표당 품목 1개 이상
- 품목명 필수
- 수량 1 이상
- 단가 0 이상
- 없는 거래처·난 묶음·전표: `404 NOT_FOUND`
- 잘못된 요청값: `400 VALIDATION_ERROR`

백엔드 테스트는 거래처 생성·검색, 전표 생성, 금액 계산, 목록·상세·출력 조회, 잘못된 요청을 검증한다.

## 8. 현재 미구현·제약

- 판매 전표 수정·삭제
- 입금·판매 상태 변경 API
- 거래처 수정·삭제
- 전표 복사
- 엑셀 다운로드
- 목록 페이지네이션
- 부분 입금 금액 관리
- 공급가액·부가세 입력 및 세금 정책
- 판매 취소·출고 이력
- 출고 완료 시 난 묶음 수량 자동 차감
- 품목 선택용 품종·난 묶음 검색
- 거래처별 월 정산표
- A5 다중 페이지 분할

기존 명세에 있는 `PATCH /api/sales-slips/{salesSlipId}/status`는 현재 구현되지 않았다.

---

## 9. 후속 구현 우선순위

1. 전표 상태 변경과 상태 이력
2. 전표 수정·취소 정책
3. 실제 입금액과 부분 입금 관리
4. 엑셀 다운로드와 페이지네이션
5. 품종·난 묶음 검색 연결
6. 출고 완료와 재고 차감 트랜잭션

재고 차감 도입 전 취소·복구·이력 정책을 먼저 확정해야 한다.
