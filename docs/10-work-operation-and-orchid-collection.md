# 작업 실행과 난 그룹 정책

## 1. 결정 상태

`2026-07-20` 기준 결정이다. 운영 작업과 난 묶음 이력은 `WorkOperation`으로 통일한다. 기존 `WorkRecord` 행은 마이그레이션 원본과 감사 목적으로 보존하지만 신규 이동·입고·사용자 작업을 기록하거나 난 묶음 이력 API에 노출하지 않는다.

동 전체·자동 그룹·사용자 그룹·직접 다중 선택 기간 작업, 월간 캘린더, 작업 상태와 그룹 관리를 구현했다. 대상 새로고침은 후속 단계다.
API는 `WORK_OPERATION_V2_ENABLED=true`일 때만 노출하며 기본값은 활성화다. 비활성화하면 기존 수동 `WorkRecord` 작성 흐름으로 복귀한다.

작업 관리 화면은 `작업 목록`, `캘린더`, `작업 이력` 탭으로 나눈다. 신규 작업 등록과 진행 관리는 작업 목록에서, 월간 일정 관리는 캘린더에서 처리한다. 작업 목록은 계획·진행 중·일시중지 작업과 농장 기준 오늘 상태가 변경된 작업만 기본 표시한다. 완료·취소·보정된 `WorkOperation`은 작업 이력에서 확인한다.

작업 목록과 작업 이력에서 작업을 선택하면 작업 유형별 입력 내용, 작업자, 메모와 대상별 실행 현황을 함께 표시한다. 농약 작업은 약제명·희석 배수·사용량을 작업 상세에서 확인한다.

분갈이·분주·합식의 일부 실행에서는 작업 수량 비율을 기준으로 원본 난 묶음 배치의 뒤쪽 구간을 자동 제안한다. 배치 선택기는 원본 난 묶음 위치를 별도 색상으로 표시한다. 사용자는 실제로 비운 뒤쪽 칸을 수정할 수 있고, 선택한 구간만큼 원본 배치가 앞쪽으로 줄어든 뒤 결과 난 묶음 배치에 재사용된다.

분갈이·분주·합식 결과 난 묶음은 작업 전 원본 상태를 이어받는다. 원본 수량을 모두 사용해 원본이 `종료`되더라도 일반 결과가 `종료` 상태를 상속하지 않는다. 분주 후보와 별도 보관 결과만 각각 `분주 예정`, `별도 보관` 상태를 사용한다.

## 2. 대상 확정

- 그룹과 상위 범위는 대상 선택 조건이다.
- 저장 시 활성 난 묶음과 수량이 1 이상인 난 묶음을 실제 대상으로 확정한다.
- 종료, 폐기, 판매 완료 상태는 제외한다.
- 작업 시작 전 새로고침은 후속 단계에서 제공한다.
- 작업 시작 후 대상은 자동 변경하지 않는다.
- 작업 당시 품종, 년생, 화분 표시값, 수량, 위치를 스냅샷으로 보존한다.
- 자동 그룹은 그룹 키, 사용자 그룹은 그룹 ID, 직접 선택은 난 묶음 ID 목록을 조건 스냅샷으로 함께 보존한다.

## 3. 자동·사용자 그룹

- 자동 그룹 기준은 `variety_id + 현재 년생 + pot_size_code`다.
- 자동 그룹은 영속 엔티티를 만들지 않고 조회 시 계산한다.
- 화분 크기 코드는 `POT_2`, `POT_2_5`, `POT_3`, `POT_3_5`, `POT_4`, `POT_4_5`, `POT_5`, `POT_6`, `HANGING`, `ETC`를 사용한다.
- 미지정은 `UNSPECIFIED`, 기존 값 중 안전하게 매핑할 수 없는 값은 `UNMAPPED`로 보존하고 자동 그룹에서 제외한다.
- `UNMAPPED` 검수 목록은 `scripts/data-audit/unmapped-pot-sizes.sql`로 조회한다.
- 자동 그룹은 조회 시 계산하며 수량 0, 종료·폐기·판매 완료, `UNMAPPED` 난 묶음을 제외한다.
- 자동 그룹 키는 조회 조건이며 난 묶음 속성이 변경되면 자동으로 새 그룹 결과에 반영된다.
- 사용자 그룹 소속은 위치와 속성이 바뀌어도 유지한다.
- 하나의 난 묶음은 여러 사용자 그룹에 속할 수 있고, 사용자가 해제하기 전까지 소속 이력을 보존한다.
- 사용자 그룹은 삭제하지 않고 보관하며, 보관된 그룹의 소속은 변경하지 않는다.
- 구조 변경 결과 난 묶음은 사용자 그룹을 기본 상속하지 않는다. 완료 시 사용자가 선택한다.

## 4. 상태와 전파

- 전체 상태: `PLANNED`, `IN_PROGRESS`, `PAUSED`, `COMPLETED`, `CANCELED`, `CORRECTED`.
- 대상 상태: `PENDING`, `IN_PROGRESS`, `COMPLETED`, `SKIPPED`, `CANCELED`, `FAILED`.
- 전체 작업은 `PLANNED → IN_PROGRESS ↔ PAUSED → COMPLETED` 흐름을 제공한다.
- 대상은 `PENDING → IN_PROGRESS → COMPLETED` 또는 `SKIPPED`로 처리한다.
- 모든 대상이 완료 또는 건너뜀 상태가 되면 전체 작업은 마지막 실행 완료일 기준으로 자동 완료된다.
- 일부 대상만 완료된 전체 작업 취소는 완료된 대상과 효과를 유지하고, 미완료 대상만 취소 상태로 닫는다.
- 모든 대상이 닫힌 진행률 100% 작업은 일시중지, 추가 실행 등록, 취소 대상이 아니다.
- 상위 작업을 난 묶음별 작업 행으로 복제하지 않는다. `WorkOperationTarget` 연결로 조회한다.
- 조회 중복 기준은 `workOperationId + orchidGroupId`다.

## 5. 구조 변경과 수량

- 사용자 정의 작업 유형은 기록형만 허용한다.
- 난 묶음 생성·종료·수량·위치 변경은 시스템 작업 유형만 수행한다.
- 분갈이 전 원본 수량은 `투입 수량 + 원본 잔여 수량`이고, 투입 수량은 `결과 수량 합계 + 손실 수량`과 같아야 한다.
- 결과 난 묶음은 원본을 덮어쓰지 않고 계보로 연결한다.
- 완료된 구조 변경 작업은 삭제하거나 일반 취소하지 않고 보정 작업을 연결한다.
- 보정 관계는 원본 작업과 별도 `CORRECTION` 작업, 사유를 보존하고 원본을 `CORRECTED` 상태로 표시한다.
- 보정 효과는 원본 작업이 만든 결과의 수량·상태만 변경하고 전후 값을 감사 상세로 보존한다.
- 후속 작업·이동·판매·입고 연결과 예약 수량을 침해하는 보정은 허용하지 않는다.
- 새 결과 생성이나 기존 계보 교체는 현재 보정 효과 범위에 포함하지 않는다.

## 6. 기존 이력

- 기존 `work_records`는 마이그레이션 원본과 감사 목적으로 보존하되 운영 이력 조회에는 사용하지 않는다.
- 기능 활성화 시 사용자 수동 작업 입력은 즉시 완료 `WorkOperation`으로 저장하고 기존 `WorkRecord` 신규 작성을 거부한다.
- 직접 위치 이동과 입고를 포함한 신규 시스템 작업도 `WorkOperation`으로 기록한다.
- 난 묶음 이력 API는 `WorkOperationTarget`과 `WorkEffectOrchidGroup` 연결만 조회하고 `WorkRecord`를 반환하지 않는다.
- V8에서 변환된 기존 이력은 변환 결과인 `WorkOperation`을 통해 조회한다.

## 7. 작업 모듈 내부 구조

- `WorkOperationPlanService`: 대상 미리보기, 계획·일괄 계획·즉시 완료 기록 생성
- `WorkOperationProgressService`: 전체·대상 상태 전이와 자동 완료
- `WorkOperationQueryService`: 상세·목록·난 묶음 이력 조회와 응답 조립
- `StructureChangeExecutionService`: 분갈이·분주·합식 회차 실행과 멱등성·잔여 수량 검증
- `InboundPottingPlanService`: 포트 후보 조회와 품종별 계획 생성
- `ImmediateWorkExecutionService`: 대상 단건·대상 없는 시스템 작업 즉시 실행
- `WorkEffectProcessor`: handler 선택과 실행
- `WorkEffectStore`: 효과·원본/결과 난 묶음 연결 저장과 효과 키 멱등성 확인

분갈이·분주·합식의 공통 N:M 변환은 `StructureChangeExecutor`가 처리하고 `RepotStrategy`, `DivideStrategy`, `MergeStrategy`가 작업별 규칙과 계보 유형을 제공한다. 포트 작업은 입고 기록이 원본이므로 `InboundPottingExecutor`의 별도 흐름을 유지한다.

작업 완료일 기본값은 `Asia/Seoul` 업무일자를 사용한다. 상태 변경 시각, 효과 적용 시각과 대상
스냅샷 시각은 UTC로 저장하고 API 응답에서만 `Asia/Seoul`로 변환한다. 테스트에서는 고정
`Clock`으로 저장·응답 시간 규칙을 검증한다.
