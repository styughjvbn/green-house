# 요구사항 정의서 v2.2

## 1. 기능 요구사항

### 1.1 농장 구조 관리

- 시스템은 총 15개 동을 관리할 수 있어야 한다.
- 각 동은 3개의 물리 배드를 가진다.
- 한 동 내부의 물리 배드는 실제 농장 구조와 동일하게 왼쪽에서 오른쪽으로 1배드, 2배드, 3배드 순서로 배치되어야 한다.
- 각 물리 배드는 중앙 관수 라인과 세로 철사 구조를 기준으로 좌/우 논리 구역처럼 활용될 수 있어야 한다.
- 초기 실행 시 1동부터 15동까지, 각 동마다 1번부터 3번까지의 물리 배드를 생성할 수 있어야 한다.
- 초기 실행 시 각 물리 배드마다 좌/우 기본 논리 구역을 생성할 수 있어야 한다.
- MVP 기준 기본 논리 구역 수는 90개다.
  - 15동 × 3물리 배드 × 2논리 구역
- 사용자는 동, 물리 배드, 논리 구역을 조회할 수 있어야 한다.
- 각 동, 물리 배드, 논리 구역은 메모를 가질 수 있다.
- 물리 배드는 실제 농장 규격 정보를 가질 수 있어야 한다.
  - 예: 대략적인 길이, 폭, 철사 줄 수, 받침대 간격 등
- 단, MVP에서는 센티미터 단위 정밀 좌표 입력을 필수로 요구하지 않는다.

### 1.2 논리 구역 관리

- 난 묶음은 물리 배드가 아니라 논리 구역에 배치되어야 한다.
- MVP에서는 기본 논리 구역을 시스템이 자동 생성한다.
- MVP에서는 사용자가 구역을 자유롭게 병합/분할하지 않는다.
- MVP 이후에는 사용자가 논리 구역을 병합, 분할, 이름 변경, 비활성화할 수 있어야 한다.
- 행잉 분화는 MVP 이후 별도 구역 유형으로 확장할 수 있어야 한다.
- 논리 구역은 농장맵에서 선택 가능한 단위여야 한다.

### 1.3 농장 현황 조회

- 사용자는 농장 현황 화면에서 전체 농장맵을 볼 수 있어야 한다.
- 농장 현황 화면은 15개 동 전체 상태 확인과 동 선택을 위한 통합 조회 화면이어야 한다.
- 농장 현황 화면에서는 난 묶음 생성, 수정, 삭제, 드래그 이동을 직접 수행하지 않는다.
- 사용자는 전체 농장맵에서 동을 클릭할 수 있어야 한다.
- 사용자가 동을 클릭하면 해당 동에 존재하는 모든 난 묶음 목록을 볼 수 있어야 한다.
- 사용자는 줌인하여 물리 배드가 보이는 단계로 이동할 수 있어야 한다.
- 사용자가 물리 배드를 클릭하면 해당 배드에 존재하는 모든 난 묶음 목록을 볼 수 있어야 한다.
- 사용자는 더 줌인하여 논리 구역이 보이는 단계로 이동할 수 있어야 한다.
- 사용자가 논리 구역을 클릭하면 해당 구역에 존재하는 난 묶음 목록과 최근 작업 요약을 볼 수 있어야 한다.
- 선택된 동/물리 배드/논리 구역은 시각적으로 강조되어야 한다.
- 농장 현황 화면의 우측 패널은 선택 대상의 범위에 맞게 목록을 표시해야 한다.

### 1.4 난 묶음 관리

- 사용자는 난 묶음 관리 화면에서 특정 논리 구역에 난 묶음을 등록할 수 있어야 한다.
- 난 묶음 관리 화면은 선택한 동의 상세맵을 중심으로 구성한다.
- 난 묶음 관리 화면에서는 선택한 동의 3개 물리 배드를 실제 농장과 동일하게 왼쪽에서 오른쪽으로 표시해야 한다.
- 하나의 논리 구역에는 여러 난 묶음이 존재할 수 있어야 한다.
- 난 묶음은 품종명, 속, 수량, 화분 크기, 연차, 상태, 메모를 가진다.
- 난 묶음은 논리 구역 안에서 표시 순서를 가져야 한다.
- 사용자는 난 묶음을 수정하거나 삭제할 수 있어야 한다.
- 사용자는 동/물리 배드/논리 구역/품종/상태 기준으로 난 묶음을 조회할 수 있어야 한다.
- MVP에서는 난 묶음 전체 이동을 지원한다.
- MVP 이후에는 난 묶음 일부 수량 이동과 자동 분할을 지원할 수 있어야 한다.

### 1.5 맵 기반 난 묶음 생성 및 이동

- 난 묶음 생성, 수정, 삭제, 이동은 기본적으로 난 묶음 관리 화면에서 수행한다.
- 사용자는 난 묶음 관리 화면에서 논리 구역을 선택한 뒤 난 묶음을 추가할 수 있어야 한다.
- 사용자는 같은 동 내부에서는 난 묶음 블록 UI를 드래그 앤 드롭하여 다른 논리 구역으로 이동할 수 있어야 한다.
- 사용자는 다른 동으로 난 묶음을 이동할 때 목적 동, 물리 배드, 논리 구역을 단계적으로 선택할 수 있어야 한다.
- 다른 동 이동은 드래그만으로 처리하지 않고 목적지 선택 방식을 기본으로 한다.
- 난 묶음 이동은 기본적으로 전체 이동으로 처리한다.
- 난 묶음 이동 시 사용자의 확인을 받아야 한다.
- 난 묶음 이동 시 위치 이동 작업 이력이 자동 생성되어야 한다.
- 실수 방지를 위해 보기 모드와 배치 수정 모드를 분리해야 한다.
- 이동 직후에는 되돌리기 기능을 제공하는 것이 권장된다.

### 1.6 작업 이력 관리

- 사용자는 농약, 비료, 분갈이, 상태 기록, 일반 메모 작업 이력을 등록할 수 있어야 한다.
- 사용자는 잎 정리, 잡초 정리, 단화/꽃 정리 작업 이력을 등록할 수 있어야 한다.
- 작업 대상은 전체 농장, 특정 동, 특정 물리 배드, 특정 논리 구역, 특정 난 묶음 중 하나일 수 있어야 한다.
- 작업 이력은 작업일, 작업 유형, 대상, 약제명 또는 비료명, 농도 또는 희석 배수, 작업자, 메모를 포함할 수 있어야 한다.
- 시스템은 구역 또는 배드별 최근 농약일, 최근 비료일, 최근 분갈이일을 계산해서 보여줘야 한다.
- 사용자는 최근 작업일을 직접 수정하지 않고 작업 이력을 통해 관리해야 한다.
- 난 묶음 이동 시 위치 이동 작업 이력이 자동 생성되어야 한다.
- MVP 이후 사용자가 작업 유형을 직접 추가/수정/비활성화할 수 있어야 한다.

### 1.7 농장맵 UI

- MVP 농장맵은 정밀 CAD 지도가 아니라 관리용 그리드로 구현한다.
- 농장맵은 화면 목적에 따라 농장 현황용 전체맵과 난 묶음 관리용 상세맵으로 역할을 나눈다.
- 농장 현황용 전체맵은 15개 동 전체 상태와 선택 범위의 난 묶음 목록을 조회하는 데 사용한다.
- 난 묶음 관리용 상세맵은 선택한 동 안에서 난 묶음 생성, 수정, 삭제, 이동을 수행하는 데 사용한다.
- 한 동 내부의 물리 배드는 실제 농장과 동일하게 왼쪽에서 오른쪽으로 표시되어야 한다.
- 각 물리 배드는 좌/우 기본 논리 구역으로 표시되어야 한다.
- 가시성 확보를 위해 난 묶음 관리용 상세맵에서는 실제 방향 보기, 회전 보기, 배드별 보기 중 하나 이상을 제공할 수 있다.
- 데이터 기준 방향은 항상 실제 농장 방향을 따른다.

### 1.8 판매 전표 관리

- 사용자는 거래처를 등록할 수 있어야 한다.
- 사용자는 판매 전표를 생성할 수 있어야 한다.
- 판매 전표에는 여러 판매 품목이 포함될 수 있어야 한다.
- 판매 품목은 상품명 또는 품종명, 속, 규격, 수량, 단가, 금액, 메모를 포함할 수 있어야 한다.
- 금액은 수량과 단가를 기준으로 자동 계산되어야 한다.
- 판매 전표의 총 금액은 판매 품목 금액 합계로 계산되어야 한다.
- 입금 상태를 관리할 수 있어야 한다.
- 판매 전표를 A5 기준으로 출력할 수 있어야 한다.
- MVP 이후에는 판매 품목이 난 묶음과 연결될 수 있어야 한다.
- MVP 이후에는 판매 완료 또는 출고 완료 시 난 묶음 수량이 자동 차감될 수 있어야 한다.

### 1.9 출력 기능

- 판매 전표를 A5 기준으로 출력할 수 있어야 한다.
- 구역별 관리표 또는 배드별 관리표를 A5 기준으로 출력할 수 있어야 한다.
- 작업 이력 목록을 A5 기준으로 출력할 수 있어야 한다.
- 출력물은 부모님 세대 사용자가 읽기 쉽도록 글자가 작지 않아야 한다.
- A5 한 장에 너무 많은 정보를 넣지 않아야 한다.
- 필요한 경우 A4 출력 옵션은 MVP 이후 추가할 수 있다.

## 2. 비기능 요구사항

### 2.1 사용성

- 버튼과 글자는 크게 표시한다.
- 주요 입력은 10초 안에 가능하도록 단순화한다.
- 현장에서 자주 사용하는 작업은 클릭 수를 최소화한다.
- 키보드 입력을 줄이고 마우스/터치 기반 조작을 우선한다.
- 화면이 복잡해지지 않도록 한다.
- PC와 태블릿 사용을 우선 고려한다.
- 조회 화면과 작업 화면을 분리하여 실수로 데이터가 변경되지 않도록 한다.
- 난 묶음 관리 화면에서 데이터 변경 작업에는 확인 또는 되돌리기 장치를 둔다.

### 2.2 안정성

- 판매 전표, 작업 이력, 난 묶음 데이터는 장기간 안정적으로 보관되어야 한다.
- 데이터 삭제는 신중하게 처리해야 한다.
- 가능하면 삭제 대신 비활성화 또는 상태 변경 방식을 고려한다.
- 난 묶음 이동, 판매 전표 취소, 출고 처리 등 수량에 영향을 주는 작업은 이력을 남겨야 한다.

### 2.3 확장성

- 초기에는 난 묶음 단위 관리로 시작한다.
- 향후 고가품, 모주, 문제 개체는 개별 화분 단위로 확장할 수 있어야 한다.
- 향후 사진 기록, QR 코드, IoT 센서, AI 진단 기능을 추가할 수 있도록 구조를 막지 않는다.
- 향후 논리 구역 병합/분할, 행잉 구역, 15구/20구 판 단위 관리, 개별 화분 위치 관리를 추가할 수 있도록 한다.
- 작업 유형은 기본값을 제공하되, 장기적으로 사용자 정의 작업 유형을 추가할 수 있어야 한다.

### 2.4 운영성

- Docker Compose로 로컬 개발 환경을 쉽게 실행할 수 있어야 한다.
- 운영 DB는 PostgreSQL을 기본으로 한다.
- 운영 데이터 백업 방침을 마련해야 한다.
- 농장 내부 서버 또는 미니 PC 배포를 고려한다.

## 3. 데이터 보존 요구사항

- 작업 이력은 삭제하지 않는 것을 기본 원칙으로 한다.
- 난 묶음 이동 이력은 위치 변경의 근거로 보존한다.
- 판매 전표는 추후 정산과 거래 확인에 필요하므로 장기 보존한다.
- 판매 수량 차감 기능이 추가될 경우 전표, 출고, 재고 변동 이력의 연결이 깨지지 않도록 관리한다.
- 사진 기록 기능이 추가될 경우 파일과 DB 메타데이터의 연결이 깨지지 않도록 관리한다.
