# 난 농장 관리 시스템 문서 인덱스 v2.2

## 문서 목적

이 문서 모음은 비닐하우스 난 농장을 관리하기 위한 웹 기반 농장 경영 시스템 개발 문서 집합이다.

본 프로젝트는 판매 전표, 농장 위치, 난 묶음, 농약·비료·분갈이·정리 작업 이력을 디지털화하고, 현장 작업자가 이해하기 쉬운 농장맵과 출력물을 제공하는 것을 목표로 한다.

v2 문서에서는 기존의 `동 → 배드 → 난 묶음` 구조를 더 실제 농장 구조에 맞게 보정한다. 실제 농장은 1동에 3개의 물리 배드가 있으나, 각 물리 배드는 중앙 관수 라인과 철사 구조를 기준으로 좌/우 논리 구역처럼 활용된다. 따라서 시스템은 `동 → 물리 배드 → 논리 구역 → 난 묶음` 구조를 기본으로 한다.

v2.2 문서에서는 화면 역할을 더 명확히 분리한다. `농장 현황`은 전체 농장맵을 통한 통합 조회 화면이고, `난 묶음 관리`는 선택한 동의 상세맵을 이용해 난 묶음을 생성·수정·이동하는 작업 화면이다.

---

## v2.2 주요 변경사항

| 구분 | 변경 내용 |
|---|---|
| 화면 역할 | 농장 현황은 조회 화면, 난 묶음 관리는 작업 화면으로 분리 |
| 농장 현황 | 전체 맵에서 동/배드/논리 구역을 클릭해 선택 범위의 난 묶음 목록 조회 |
| 줌 조회 | 전체 농장 보기 → 동 보기 → 물리 배드 보기 → 논리 구역 보기 흐름 정의 |
| 난 묶음 관리 | 선택한 동의 상세맵에서 난 묶음 생성·수정·삭제·이동 수행 |
| 동 상세맵 | 물리 배드 1, 2, 3을 실제 농장과 동일하게 좌→우 배치 |
| 이동 UX | 같은 동 내부는 드래그 앤 드롭, 다른 동 이동은 목적지 선택 방식 |
| API | 농장 현황 조회용 `/api/farm-status/*` API 추가 |
| 개발 계획 | 농장 현황 맵 구현과 난 묶음 관리 상세맵 구현을 별도 단계로 분리 |

## 기존 v2 주요 변경사항 유지

| 구분 | 변경 내용 |
|---|---|
| 농장 구조 | 1동 3배드 구조에서 물리 배드 3개 + 기본 논리 구역 6개 구조로 확장 |
| 구역 관리 | MVP에서는 기본 논리 구역 자동 생성, MVP 이후 사용자 정의 병합/분할 지원 |
| 난 묶음 배치 | 난 묶음은 배드가 아니라 논리 구역에 배치되며, 구역 안에서 순서를 가진다 |
| 이동 이력 | 난 묶음 이동 시 위치 이동 작업 이력을 자동 생성 |
| 작업 유형 | 잎 정리, 잡초 정리, 단화/꽃 정리를 기본 작업 유형에 추가 |
| 작업 유형 확장 | MVP 이후 사용자 정의 작업 유형을 추가할 수 있는 구조 고려 |
| 판매 전표 | MVP는 전표 생성/출력 중심, 이후 난 묶음 수량 자동 차감과 연동 |
| 출력 | A4가 아니라 A5를 기본 출력 규격으로 변경 |

## 현재 MVP 구현 상태

현재 구현은 다음 범위까지 완료된 상태를 기준으로 한다.

- 대시보드 요약 화면
- 농장 현황 조회 화면
- 난 묶음 관리 상세맵
- 난 묶음 생성·수정·삭제
- 같은 동 내부 드래그 이동
- 다른 동 목적지 선택 이동
- 이동 시 위치 이동 작업 이력 자동 생성
- 작업 이력 등록/조회 화면
- 거래처 등록/조회
- 판매 전표 생성/목록/상세 화면
- 판매 전표 A5 출력 화면
- 서비스 글자 크기 설정

다음 기능은 MVP 이후 범위로 둔다.

- 논리 구역 병합/분할
- 난 묶음 일부 수량 이동과 자동 분할
- 여러 난 묶음 일괄 이동
- 판매 전표 수정/삭제 및 상태 변경 API
- 판매 완료 또는 출고 완료 시 난 묶음 수량 자동 차감
- 작업 이력 수정/삭제
- 작업 이력 A5 출력
- 논리 구역/물리 배드 관리표 출력
- 사용자 인증/권한

## 문서 작성 원칙

- 요구사항 문서에는 “무엇을 해야 하는지”를 쓴다.
- 도메인 문서에는 “업무 개념과 규칙”을 쓴다.
- 기능정의서에는 “사용자가 어떤 기능을 사용할 수 있어야 하는지”를 쓴다.
- UI/UX 문서에는 “사용자가 어떻게 보고 조작할지”를 쓴다.
- API 문서에는 “프론트엔드와 백엔드가 어떻게 통신할지”를 쓴다.
- DB 문서에는 “데이터를 어떻게 저장할지”를 쓴다.
- 아키텍처 문서에는 “코드를 어떤 구조로 나눌지”를 쓴다.
- 배포 문서에는 “어디서 어떻게 실행하고 백업할지”를 쓴다.
