# 판매·경매·정산 관리

## 1. 목적

판매 관리는 단순 전표 입력뿐 아니라 다음 흐름을 함께 다룬다.

```text
일반 판매
→ 판매 전표
→ 입금 확인
→ 거래처 잔액 반영

경매 출하
→ 경매 lot
→ 경매 결과/유찰/반환 추적
→ 경매 정산
→ 입금 확인
```

## 2. 거래처 관리

거래처 유형:

- `WHOLESALE`: 도매상
- `RETAIL`: 소매상
- `AUCTION_HOUSE`: 경매장

경매장도 거래처로 관리한다. 다만 일반 거래처와 달리 경매 lot, 정산 묶음, 입금 지연일, 정산 단위 설정이 필요하다.

## 3. 일반 판매 전표

```text
BusinessPartner
 └─ SalesSlip
    └─ SalesSlipItem
```

현재 기능:

- 거래처 선택
- 판매일 입력
- 품목 추가
- 수량/단가 입력
- 금액 자동 계산
- 전표 합계 계산
- A5 출력
- 입금 확인

운영 규칙:

- 전표 생성만으로 난 묶음 수량을 자동 차감하지 않는다.
- 수량 차감은 출고 완료 또는 판매 확정 상태를 기준으로 추후 연동한다.
- 전표 삭제보다 취소 상태를 우선한다.

## 4. 경매 출하 추적

경매 데이터는 일반 판매 전표 품목에 바로 넣지 않는다.

이유:

- 출하일과 판매일이 다를 수 있다.
- 같은 lot이 유찰 후 다시 경매될 수 있다.
- 일부만 판매되고 일부는 대기 또는 반환될 수 있다.
- 경매 결과 행을 단순 합산하면 유찰 재등장 물량이 중복 계산된다.

따라서 경매는 lot 단위로 추적한다.

```text
AuctionLot
 ├─ AuctionAttempt
 │   └─ AuctionResultLine
 └─ AuctionStatusHistory
```

## 5. AuctionLot 상태

대표 상태:

- `SHIPPED`: 출하됨
- `WAITING`: 경매 대기
- `IN_PROGRESS`: 경매 진행 중
- `SOLD`: 판매 완료
- `PARTIALLY_SOLD`: 부분 판매
- `FAILED`: 유찰
- `REAUCTION_WAITING`: 재경매 대기
- `RETURN_INFERRED`: 반환 추정
- `PARTIALLY_RETURNED`: 부분 반환
- `RETURNED`: 반환 완료
- `QUANTITY_MISMATCH`: 수량 불일치
- `REVIEW_REQUIRED`: 확인 필요
- `CANCELLED`: 취소

상태 변경은 이력으로 남긴다.

## 6. 경매 매칭 기준

업무상 조회 키:

```text
출하일자 + 품종명 + 등급 + 경매장
```

단, 이 조합은 유일키로 쓰기 어렵다.

권장 매칭 단계:

| 단계 | 기준 | 처리 |
|---|---|---|
| 1차 | 출하일자 + 품종명 + 등급 + 경매장 | 자동 확정 후보 |
| 2차 | 출하일자 + 품종명 + 경매장 | 확인 필요 후보 |
| 3차 | 수량/유찰/반환 흐름 비교 | 수동 검수 |

시스템 내부 식별자는 반드시 `auction_lot.id` 같은 별도 ID를 사용한다.

## 7. 경매 정산

```text
AuctionSettlement
 └─ AuctionSettlementLine
```

정산은 경매장과 정산 기간을 기준으로 묶는다.

정산에 포함될 수 있는 대상:

- 판매 완료 lot
- 부분 판매 lot의 판매분
- 경매 결과 행
- 수수료/차감액

정산 상태 예시:

- 생성됨
- 입금 대기
- 부분 입금
- 입금 완료
- 보류

정산 재계산은 가능해야 하지만, 재계산 전후 차이가 추적 가능해야 한다.

## 8. 입금 처리

입금 처리는 `PartnerPaymentEvent`로 보존한다.

대상:

- 판매 전표
- 경매 정산
- 대상 미지정 입금

처리해야 할 상황:

- 전액 입금
- 부분 입금
- 초과 입금
- 여러 전표/정산에 나눠 매칭
- 입금자명 또는 계좌 자동 매칭
- 수동 보정

운영 규칙:

- 입금 이벤트는 삭제하지 않는다.
- 잘못 매칭한 경우 취소 또는 보정 이벤트를 남긴다.
- 자동 매칭은 확정 처리 전에 검토 상태를 둘 수 있다.

## 9. 초기 구현 방식

테이블 폭증을 막기 위해 초기에는 일부 상세 원본/보정 정보는 JSONB로 보관할 수 있다.

JSONB 후보:

- 원본 가져오기 행
- 자동 매칭 후보
- 은행 입금 원본 메타데이터
- 경매장별 비정형 정산 정보

다만 조회와 정산 기준으로 자주 쓰이는 값은 컬럼으로 둔다.

컬럼 우선 후보:

- 거래처 ID
- 경매장 ID
- 출하일
- 경매일
- lot 상태
- 출하 수량
- 판매 수량
- 반환 수량
- 정산 금액
- 입금 금액
- 입금 상태
